Orientation

How DSA / coding rounds actually work

★ high-freq

For new-grad and SDE roles at product companies (Google, Microsoft, Amazon, Atlassian, Uber, De Shaw, Sprinklr, PhonePe, and the rest of the placement circuit), the coding bar is almost entirely data structures & algorithms, and C++ is the default weapon. Know the stages:

  • Online assessment — 2–4 timed problems on HackerRank / Codility. Speed + correctness + passing hidden tests. Fast I/O and clean templates matter here.
  • DSA interviews — 1–3 rounds, 1–2 problems each, live coding. They watch you think: clarify constraints, state the brute force, optimise, state complexity, then code it cleanly and dry-run.
  • Problem-solving + CS fundamentals — follow-ups on complexity, why your data structure, OS/DBMS/OOP basics woven in.
  • LLD / machine coding (some companies) — design a system in OOP C++; less common for fresh grads than for SDE-2+.

The 6-step playbook for any problem

★ high-freq

Interviewers score the process, not just the final code. Run this loop out loud every time — it is the single biggest differentiator between candidates who pass and ones who don't.

  • 1. Clarify — input ranges, types, duplicates, sorted or not, empty input, negative numbers. Restate the problem.
  • 2. Examples — walk one small example by hand, including an edge case.
  • 3. Brute force — state it with complexity. Shows you understand the problem before optimising.
  • 4. Optimise — name the pattern (hashing, two pointers, binary search, DP). Explain the trade-off.
  • 5. Code — clean, named variables, handle edges. Talk while you type.
  • 6. Verify — dry-run your example, check edges (empty, single element, overflow), restate final time/space.

C++ for Interviews

The competitive C++ template

★ high-freqtemplate

Start every problem from a known shell so you're not wiring boilerplate under time pressure. <bits/stdc++.h> pulls in the whole standard library (GCC-only, but that's what judges run), and the two I/O lines below make cin/cout fast enough for 106+ lines.

cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);  // decouple from C stdio
    cin.tie(nullptr);                  // don't flush cout before each cin

    int n;
    cin >> n;
    vector<int> a(n);
    for (int& x : a) cin >> x;         // read by reference

    // solve...
    return 0;
}

Fast I/O — why and when

gotchawrite cold

By default C++ streams stay synchronised with C's stdio and cin is tied to cout (every read flushes output first). Both cost time. Disabling them can turn a TLE into an AC on large inputs. Caveat: after sync_with_stdio(false), don't mix scanf/printf with cin/cout.

cpp
ios_base::sync_with_stdio(false);
cin.tie(nullptr);

// Prefer "\n" over endl in loops — endl flushes every time (slow).
cout << answer << "\n";

Integer types & overflow

★ high-freqgotcha

The most common silent bug in interviews. int holds up to ~2.1×109; a product of two ints, or a sum of many, overflows quietly and gives a wrong answer with no crash. When values or intermediate results can exceed ~109, use long long, and cast before the multiply.

cpp
int a = 100000, b = 100000;
long long bad  = a * b;            // overflow: int*int computed first
long long good = (long long)a * b; // cast first → correct

// Handy limits
INT_MAX;    // 2147483647
LLONG_MAX;  // 9223372036854775807
const int INF = 1e9;              // safe "infinity" that won't overflow on +

auto, range-for & structured bindings

★ high-freq

Modern C++ (C++17) cuts noise. auto infers types, range-for iterates cleanly, and structured bindings unpack pairs/tuples/structs. Iterate by reference (auto&) to mutate or to avoid copying big objects.

cpp
vector<pair<int,int>> edges = {{1,2},{3,4}};

for (auto [u, v] : edges)          // structured binding
    cout << u << "-" << v << "\n";

for (auto& x : a) x *= 2;           // by reference → mutate in place

map<string,int> freq;
for (auto& [word, count] : freq)   // unpack map entries
    cout << word << ": " << count << "\n";

Pass by reference, not by value

★ high-freqgotcha

Passing a vector/string by value copies the whole thing — O(n) per call and a frequent cause of TLE in recursion. Pass by reference. Use const& when you only read, plain & when you mutate.

cpp
void solve(vector<int>& a);          // mutate caller's vector
int sum(const vector<int>& a);       // read-only, no copy

// In recursion this matters a lot:
void dfs(int u, vector<vector<int>>& adj, vector<bool>& seen);
// passing adj by value would copy the whole graph at every call

Lambdas

★ high-freq

Anonymous inline functions. Essential for custom comparators and for recursion-as-a-local (with an explicit function type or a generic-lambda self). [&] captures surrounding variables by reference, [=] by value, [] captures nothing.

cpp
auto add = [](int a, int b) { return a + b; };

int threshold = 5;
auto above = [&](int x) { return x > threshold; };  // captures threshold

// Recursive lambda (e.g. local DFS) via std::function
function<int(int)> fib = [&](int n) {
    return n < 2 ? n : fib(n-1) + fib(n-2);
};

Custom comparators

★ high-freqgotcha

Control sort order and ordered-container ordering. The comparator returns true if a should come before b. It must be a strict weak ordering (use <, never <=) or you risk a crash on some inputs.

cpp
// Sort: by second asc, tie-break first desc
sort(v.begin(), v.end(), [](auto& a, auto& b) {
    if (a.second != b.second) return a.second < b.second;
    return a.first > b.first;
});

// Min-heap of pairs
priority_queue<pair<int,int>, vector<pair<int,int>>, greater<>> pq;

// Ordered set with custom order
auto cmp = [](int a, int b) { return a > b; };
set<int, decltype(cmp)> s(cmp);

Complexity & Constraints

Big-O in one screen

★ high-freq

Big-O describes how runtime/space grows with input size n, ignoring constants and lower-order terms. You state it for every solution. Drop constants (O(2n)=O(n)) and keep the dominant term (O(n+n²)=O(n²)).

  • O(1) hash lookup, array index, push_back (amortised)
  • O(log n) binary search, balanced-tree / heap op
  • O(n) single pass, two pointers, BFS/DFS over a list
  • O(n log n) sorting, divide & conquer, heap-based
  • O(n²) nested loops, naive pair comparisons
  • O(2ⁿ), O(n!) subsets, permutations — only for tiny n

Read the constraints, pick the complexity

★ high-freqpattern

The input bounds tell you the intended complexity before you've solved anything. A rough budget of ~108 simple operations per second is the mental model.

  • n ≤ 10–12 → O(2ⁿ) or O(n!) — bitmask / brute force is fine
  • n ≤ ~500 → O(n³) ok (Floyd-Warshall, some DP)
  • n ≤ ~5000 → O(n²)
  • n ≤ 10⁵–10⁶ → O(n log n) or O(n)
  • n ≥ 10⁸ → O(n) or O(log n); be careful with I/O

Amortised vs worst case

gotcha

vector::push_back is O(1) amortised: occasionally it doubles capacity and copies everything (O(n)), but averaged over many pushes it's O(1). unordered_map is O(1) average but O(n) worst case under hash collisions — mention this when an interviewer probes.

Space complexity counts too

pattern

Recursion uses O(depth) stack space — a recursion n deep is O(n) memory even if it allocates nothing. An auxiliary hashmap, a DP table, or a copy of the input all add up. “Can you do it in O(1) extra space?” is a common follow-up (in-place reversal, two pointers, Floyd's cycle detection).

STL

Sequence Containers

vector — the workhorse

★ high-freqSTL

A dynamic array: contiguous memory, O(1) random access and back-insertion, the default container for almost everything. Reserve capacity when you know the size to avoid reallocations.

cpp
vector<int> v;                 // empty
vector<int> v(n, 0);           // n zeros
vector<vector<int>> grid(r, vector<int>(c, 0)); // r x c matrix

v.push_back(x);                // O(1) amortised
v.pop_back();                  // O(1)
v.back(); v.front();           // ends
v[i];                          // O(1), no bounds check
v.size(); v.empty();
v.reserve(n);                  // pre-allocate, avoids realloc
sort(v.begin(), v.end());

string

★ high-freqSTL

A vector of chars with text conveniences. Concatenate with +=, convert with to_string / stoi, build big strings incrementally (don't rebuild in a loop).

cpp
string s = "hello";
s += " world";                 // append
s.substr(0, 3);                // "hel"  (pos, len)
s.size(); s[i];
int x = stoi("42");            // string → int
string t = to_string(99);      // int → string
reverse(s.begin(), s.end());
sort(s.begin(), s.end());      // sort chars (e.g. anagram key)

pair & tuple

★ high-freqSTL

Lightweight fixed bundles. pair for two values (coordinates, edge+weight), tuple for three or more. Compared lexicographically by default — great as heap/sort elements.

cpp
pair<int,int> p = {3, 4};
p.first; p.second;
auto [x, y] = p;               // unpack

tuple<int,int,int> t = {1, 2, 3};
auto [a, b, c] = t;
get<0>(t);                     // element by index

vector<pair<int,int>> pts;
pts.push_back({x, y});

stack & queue

★ high-freqSTL

LIFO and FIFO adapters over a deque. stack for DFS, parsing, monotonic patterns; queue for BFS, level-order. Note the asymmetric names (top vs front).

cpp
stack<int> st;
st.push(1); st.top(); st.pop(); st.empty();

queue<int> q;
q.push(1); q.front(); q.back(); q.pop(); q.empty();

deque

STL

Double-ended queue: O(1) push/pop at both ends. The container behind the sliding-window-maximum (monotonic deque) pattern and any “add/remove from both sides” need.

cpp
deque<int> dq;
dq.push_back(1);  dq.push_front(2);
dq.pop_back();    dq.pop_front();
dq.front(); dq.back(); dq[i];  // random access too

array & iterators

STL

std::array<T,N> is a fixed-size stack array with STL conveniences (rarely needed over vector). Iterators are the glue every algorithm uses: begin()..end() is a half-open range, rbegin()/rend() reverse.

cpp
array<int,3> a = {1, 2, 3};

sort(v.rbegin(), v.rend());        // descending sort via reverse iters
auto it = v.begin() + 2;           // random-access iterator arithmetic
v.erase(v.begin() + i);            // erase by iterator (O(n) shift)
STL

Maps, Sets & Heaps

map vs unordered_map

★ high-freqSTL

map is a balanced BST: keys sorted, every op O(log n), supports range/ordered queries. unordered_map is a hash table: O(1) average, no ordering, O(n) worst case. Default to unordered_map for plain key→value lookups; reach for map when you need keys in order or lower_bound.

cpp
unordered_map<string,int> cnt;
cnt["a"]++;                    // inserts 0 then increments
if (cnt.count("a")) {...}      // membership
cnt.find("a");                 // iterator or end()
cnt.erase("a");

map<int,int> ordered;          // keys sorted ascending
ordered.begin()->first;        // smallest key
auto it = ordered.lower_bound(k); // first key >= k

set vs unordered_set / multiset

★ high-freqSTLgotcha

set = sorted unique keys (O(log n)); unordered_set = hashed unique keys (O(1) avg); multiset keeps duplicates and stays sorted — useful for a running multiset where you need the min/max and ordered erase.

cpp
unordered_set<int> seen;
seen.insert(x); seen.count(x); seen.erase(x);

set<int> s;                    // sorted, unique
*s.begin();                    // min     *s.rbegin() = max
s.lower_bound(k);              // first >= k

multiset<int> ms;              // sorted, duplicates kept
ms.erase(ms.find(x));          // erase ONE x (erase(x) removes all!)

priority_queue (heap)

★ high-freqSTL

A binary heap. Default is a max-heap (largest on top). For a min-heap, add greater<>. O(log n) push/pop, O(1) top. The backbone of Dijkstra, top-K, and merge-K problems.

cpp
priority_queue<int> maxh;                          // max-heap
maxh.push(x); maxh.top(); maxh.pop();

priority_queue<int, vector<int>, greater<int>> minh; // min-heap

// Min-heap of (dist, node) for Dijkstra
priority_queue<pair<int,int>, vector<pair<int,int>>, greater<>> pq;

Custom hash for pair / using a map key

gotchatemplate

The classic gotcha: unordered_map<pair<int,int>,...> won't compile — there's no default hash for pair. Either provide a hash functor, or just use map (ordered, no hash needed), or encode the pair into one long long key.

cpp
struct PairHash {
    size_t operator()(const pair<int,int>& p) const {
        return hash<long long>()((long long)p.first * 1000000007LL + p.second);
    }
};
unordered_map<pair<int,int>, int, PairHash> mp;

// Simpler escape hatches:
map<pair<int,int>, int> ordered;            // no hash required
unordered_map<long long, int> encoded;      // key = (long long)x*COLS + y
STL

Algorithms

sort & custom ordering

★ high-freqSTL

Introsort, O(n log n). The single most-used algorithm. Add a comparator lambda for custom order. stable_sort preserves relative order of equal elements when that matters.

cpp
sort(v.begin(), v.end());                       // ascending
sort(v.begin(), v.end(), greater<int>());       // descending
sort(v.begin(), v.end(), [](auto& a, auto& b){
    return a.first < b.first;                   // by field
});
stable_sort(v.begin(), v.end(), cmp);           // keeps ties' order

binary_search family

★ high-freqSTL

On a sorted range: binary_search returns bool; lower_bound returns the first element >= x; upper_bound the first > x. Subtract begin() to get an index. These save you from hand-rolling binary search.

cpp
bool found = binary_search(v.begin(), v.end(), x);

auto lo = lower_bound(v.begin(), v.end(), x);   // first >= x
auto hi = upper_bound(v.begin(), v.end(), x);   // first > x
int idx   = lo - v.begin();                     // position
int count = hi - lo;                            // # of x's in sorted v

Accumulate, count, min/max element

STL

One-liners that replace manual loops and read cleanly. Watch the accumulator type on accumulate — pass 0LL to avoid int overflow on a large sum.

cpp
long long total = accumulate(v.begin(), v.end(), 0LL);
int c   = count(v.begin(), v.end(), x);
int even = count_if(v.begin(), v.end(), [](int x){ return x%2==0; });
int mx = *max_element(v.begin(), v.end());
int mn = *min_element(v.begin(), v.end());

next_permutation & friends

STLpattern

Generate permutations in lexicographic order in-place; loops until it wraps. reverse, unique (removes consecutive dups — sort first), and fill/iota round out the toolkit.

cpp
sort(v.begin(), v.end());
do {
    // use this permutation
} while (next_permutation(v.begin(), v.end()));

reverse(v.begin(), v.end());
v.erase(unique(v.begin(), v.end()), v.end());   // dedupe a SORTED vector
iota(v.begin(), v.end(), 0);                    // 0,1,2,3,...

Bit & math built-ins

STLpattern

GCC intrinsics that are faster and shorter than hand-written loops — popular in tight solutions. Append ll for long long versions (__builtin_popcountll).

cpp
__builtin_popcount(x);   // number of set bits
__builtin_clz(x);        // leading zeros (x != 0)
__builtin_ctz(x);        // trailing zeros (x != 0)
__gcd(a, b);             // GCD (or std::gcd / std::lcm in C++17)

// Fast checks
bool isPow2 = x && !(x & (x - 1));

Arrays & Strings

Prefix sums

★ high-freqpatternwrite cold

Precompute cumulative sums once (O(n)), then answer any range-sum query in O(1). The foundation for many subarray problems. 2D variants extend the same idea to submatrix sums.

cpp
vector<long long> pre(n + 1, 0);
for (int i = 0; i < n; i++) pre[i+1] = pre[i] + a[i];

// sum of a[l..r] inclusive:
long long s = pre[r+1] - pre[l];

Two pointers

★ high-freqpatternwrite cold

Two indices moving through the array — from both ends (sorted-pair search, palindrome) or both from the left (in-place filtering, fast/slow). Turns many O(n²) scans into O(n).

cpp
// Pair summing to target in a SORTED array
int l = 0, r = n - 1;
while (l < r) {
    int sum = a[l] + a[r];
    if (sum == target) return {l, r};
    else if (sum < target) l++;     // need larger
    else r--;                       // need smaller
}

Sliding window — fixed size

★ high-freqpatternwrite cold

For “every window of length k”. Maintain a running aggregate: add the entering element, remove the leaving one. O(n) instead of O(n·k).

cpp
long long sum = 0, best = LLONG_MIN;
for (int i = 0; i < n; i++) {
    sum += a[i];
    if (i >= k) sum -= a[i - k];    // drop element leaving the window
    if (i >= k - 1) best = max(best, sum);
}

Sliding window — variable size

★ high-freqpatternwrite cold

For “longest/shortest substring/subarray satisfying X”. Expand right to include, shrink left while the window is invalid. Each index enters and leaves at most once → O(n).

cpp
// Longest substring without repeating characters
int longestUnique(string& s) {
    unordered_map<char,int> last;
    int left = 0, best = 0;
    for (int right = 0; right < (int)s.size(); right++) {
        if (last.count(s[right]) && last[s[right]] >= left)
            left = last[s[right]] + 1;     // jump past the duplicate
        last[s[right]] = right;
        best = max(best, right - left + 1);
    }
    return best;
}

Kadane's algorithm

★ high-freqwrite cold

Maximum-sum contiguous subarray in O(n). At each step decide: extend the current run, or restart from here. A canonical “must write in 2 minutes” problem.

cpp
int maxSubarray(vector<int>& a) {
    int best = a[0], cur = a[0];
    for (int i = 1; i < (int)a.size(); i++) {
        cur  = max(a[i], cur + a[i]);   // restart vs extend
        best = max(best, cur);
    }
    return best;
}

Dutch national flag (3-way partition)

patternwrite cold

Sort an array of 0/1/2 (or partition around a pivot) in one O(n) pass, O(1) space. Three pointers carve the array into low / mid / high regions.

cpp
void sortColors(vector<int>& a) {
    int low = 0, mid = 0, high = a.size() - 1;
    while (mid <= high) {
        if (a[mid] == 0) swap(a[low++], a[mid++]);
        else if (a[mid] == 1) mid++;
        else swap(a[mid], a[high--]);   // don't advance mid here
    }
}

Merge intervals

★ high-freqpatternwrite cold

Sort by start, then sweep: if the next interval overlaps the current merged one, extend; otherwise emit and start fresh. The base technique for calendar/booking problems.

cpp
vector<vector<int>> merge(vector<vector<int>>& iv) {
    sort(iv.begin(), iv.end());
    vector<vector<int>> res;
    for (auto& cur : iv) {
        if (!res.empty() && cur[0] <= res.back()[1])
            res.back()[1] = max(res.back()[1], cur[1]); // overlap
        else
            res.push_back(cur);
    }
    return res;
}
Hashing

Hashing & Frequency Maps

Frequency counting

★ high-freqpattern

The most common use of a hashmap: count occurrences in one pass, then answer questions about them. Anagrams, majority element, first unique char — all frequency-map problems.

cpp
unordered_map<int,int> freq;
for (int x : a) freq[x]++;          // x defaults to 0, then ++

// most frequent
int bestKey = -1, bestCnt = 0;
for (auto& [k, c] : freq)
    if (c > bestCnt) { bestCnt = c; bestKey = k; }

Two-sum pattern (complement lookup)

★ high-freqpatternwrite cold

Whenever you need a pair/relationship and the array isn't sorted, store what you've seen in a hashmap and look up the complement in O(1). Turns O(n²) into O(n).

cpp
vector<int> twoSum(vector<int>& a, int target) {
    unordered_map<int,int> seen;        // value → index
    for (int i = 0; i < (int)a.size(); i++) {
        int need = target - a[i];
        if (seen.count(need)) return {seen[need], i};
        seen[a[i]] = i;
    }
    return {};
}

Grouping by a key

pattern

Bucket items under a computed signature. Anagram grouping uses the sorted string as the key; “group by remainder / by row” follow the same shape.

cpp
unordered_map<string, vector<string>> groups;
for (auto& w : words) {
    string key = w;
    sort(key.begin(), key.end());   // anagrams share a sorted key
    groups[key].push_back(w);
}

When NOT to trust O(1)

gotcha

unordered_map is O(1) average but adversarial inputs can force collisions → O(n) per op, O(n²) overall (this has caused real TLEs on Codeforces). If worst case matters or keys are ordered, use map (O(log n) guaranteed). For pairs, a good custom hash also helps.

Pointers

Linked Lists

Node definition & the dummy-head trick

★ high-freqtemplate

A dummy node before the head removes the “what if we modify the first element” special case — you always have a stable node to attach to. Use it whenever the head can change.

cpp
struct ListNode {
    int val;
    ListNode* next;
    ListNode(int x) : val(x), next(nullptr) {}
};

// dummy head pattern
ListNode dummy(0);
ListNode* tail = &dummy;
// ... tail->next = node; tail = tail->next;
return dummy.next;

Reverse a linked list

★ high-freqwrite cold

The single most-asked linked-list question. Walk the list, flipping each next pointer, tracking prev. O(n) time, O(1) space. Know both iterative and recursive.

cpp
ListNode* reverse(ListNode* head) {
    ListNode* prev = nullptr;
    while (head) {
        ListNode* next = head->next;  // save
        head->next = prev;            // flip
        prev = head;                  // advance
        head = next;
    }
    return prev;                      // new head
}

Fast & slow pointers (Floyd)

★ high-freqpatternwrite cold

One pointer moves twice as fast. Detects cycles (they meet), finds the middle (slow lands there when fast hits the end), and finds the k-th from end. O(1) space — the answer to “can you avoid the hashset?”.

cpp
bool hasCycle(ListNode* head) {
    ListNode *slow = head, *fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
        if (slow == fast) return true;   // pointers met → cycle
    }
    return false;
}

ListNode* middle(ListNode* head) {       // slow = middle at the end
    ListNode *slow = head, *fast = head;
    while (fast && fast->next) { slow = slow->next; fast = fast->next->next; }
    return slow;
}

Merge two sorted lists

★ high-freqwrite cold

Classic merge step (also the merge in merge-sort on lists). Dummy head + compare-and-splice. Extends to merging K lists with a heap.

cpp
ListNode* merge(ListNode* a, ListNode* b) {
    ListNode dummy(0); ListNode* tail = &dummy;
    while (a && b) {
        if (a->val <= b->val) { tail->next = a; a = a->next; }
        else                  { tail->next = b; b = b->next; }
        tail = tail->next;
    }
    tail->next = a ? a : b;          // attach the remainder
    return dummy.next;
}
Linear

Stacks & Queues

Valid parentheses

★ high-freqwrite cold

Push opening brackets, pop and match on closing. The canonical stack warm-up; the matching-map keeps it clean and extends to multiple bracket types.

cpp
bool isValid(string s) {
    stack<char> st;
    unordered_map<char,char> match = {{')','('},{']','['},{'}','{'}};
    for (char c : s) {
        if (c=='(' || c=='[' || c=='{') st.push(c);
        else {
            if (st.empty() || st.top() != match[c]) return false;
            st.pop();
        }
    }
    return st.empty();
}

Monotonic stack (next greater element)

★ high-freqpatternwrite cold

Keep the stack in increasing/decreasing order; when a new element breaks the order, pop and resolve those elements. Solves “next greater/smaller”, “daily temperatures”, “largest rectangle in histogram” — all in O(n).

cpp
vector<int> nextGreater(vector<int>& a) {
    int n = a.size();
    vector<int> res(n, -1);
    stack<int> st;                    // indices, values decreasing
    for (int i = 0; i < n; i++) {
        while (!st.empty() && a[st.top()] < a[i]) {
            res[st.top()] = a[i];     // a[i] is the next greater
            st.pop();
        }
        st.push(i);
    }
    return res;
}

Min stack (O(1) minimum)

Support push/pop/top and getMin all in O(1). Trick: a second stack tracks the minimum seen so far at each level.

        mn.push(mn.empty() ? x : min(x, mn.top()));

Sliding window maximum (monotonic deque)

patternwrite cold

Maintain a deque of indices whose values are decreasing; the front is always the window max. Each index is pushed and popped once → O(n) for the whole sweep.

cpp
vector<int> maxWindow(vector<int>& a, int k) {
    deque<int> dq;                    // indices, values decreasing
    vector<int> res;
    for (int i = 0; i < (int)a.size(); i++) {
        if (!dq.empty() && dq.front() == i - k) dq.pop_front(); // out of window
        while (!dq.empty() && a[dq.back()] <= a[i]) dq.pop_back();
        dq.push_back(i);
        if (i >= k - 1) res.push_back(a[dq.front()]);
    }
    return res;
}

Iterative in-order (explicit stack)

The “do it without recursion” follow-up. Push lefts, then process and turn right. Same idea drives BST iterators and Morris traversal (O(1) space).

vector<int> inorder(TreeNode* root) {

    TreeNode* cur = root;

        cur = st.top(); st.pop();

        cur = cur->right;

Height & diameter

★ high-freqpattern

Lowest common ancestor

★ high-freqpattern

In a plain binary tree: recurse; if a node is one of the targets return it, and the node where the two sides both return non-null is the LCA. In a BST it's simpler — walk down while both targets are on the same side.

TreeNode* lca(TreeNode* root, TreeNode* p, TreeNode* q) {

    TreeNode* L = lca(root->left,  p, q);
    TreeNode* R = lca(root->right, p, q);
    if (L && R) return root;          // p and q split here
    return L ? L : R;
}

Top-K elements

★ high-freqpatternwrite cold

For “k largest”, keep a min-heap of size k: push, and pop when size exceeds k so the smallest falls out. O(n log k), better than sorting when k ≪ n. (For k smallest, use a max-heap.)

Median from a data stream (two heaps)

patternwrite cold

Balance a max-heap (lower half) and a min-heap (upper half) so their sizes differ by at most one. The median is then a top (odd count) or the average of both tops (even). O(log n) per insert.

    hi.push(lo.top()); lo.pop();          // push largest of lower to upper
    if (hi.size() > lo.size()) { lo.push(hi.top()); hi.pop(); } // rebalance
}

                                 : (lo.top() + hi.top()) / 2.0;

BFS (shortest path in unweighted graph)

★ high-freqpatternwrite cold

Layer-by-layer from the source; the first time you reach a node is via a shortest path (fewest edges). Mark visited when enqueuing, not dequeuing, to avoid duplicates.

vector<int> bfs(int src, vector<vector<int>>& adj, int n) {

Cycle detection

pattern

Undirected: a visited neighbour that isn't your parent means a cycle. Directed: track nodes in the current recursion stack — hitting one of those is a back edge (cycle). Topological sort failing is another directed-cycle signal.

    color[u] = 1;
    for (int v : adj[u]) {
        if (color[v] == 1) return true;           // back edge
        if (color[v] == 0 && hasCycle(v, adj, color)) return true;
    }

    return false;

Topological sort (Kahn's / BFS)

★ high-freqpatternwrite cold

Linear ordering of a DAG so every edge points forward. Repeatedly emit zero-in-degree nodes. If you can't emit all V nodes, the graph has a cycle. Used for build order, course schedule, dependency resolution.

cpp
vector<int> topoSort(int n, vector<vector<int>>& adj) {
    vector<int> indeg(n, 0), order;
    for (int u = 0; u < n; u++)
        for (int v : adj[u]) indeg[v]++;
    queue<int> q;
    for (int i = 0; i < n; i++) if (indeg[i] == 0) q.push(i);
    while (!q.empty()) {
        int u = q.front(); q.pop();
        order.push_back(u);
        for (int v : adj[u]) if (--indeg[v] == 0) q.push(v);
    }
    return (int)order.size() == n ? order : vector<int>{}; // empty = cycle
}

Dijkstra (shortest path, non-negative weights)

★ high-freqpatternwrite cold

BFS generalised to weighted edges using a min-heap of (distance, node). Pop the closest unsettled node, relax its edges. O((V+E) log V). Skip stale heap entries with the d > dist[u] check. Doesn't work with negative edges — that's Bellman-Ford.

cpp
vector<long long> dijkstra(int src, vector<vector<pair<int,int>>>& adj, int n) {
    vector<long long> dist(n, LLONG_MAX);
    priority_queue<pair<long long,int>,
                   vector<pair<long long,int>>, greater<>> pq;
    dist[src] = 0; pq.push({0, src});
    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (d > dist[u]) continue;            // stale entry
        for (auto [v, w] : adj[u])
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                pq.push({dist[v], v});
            }
    }
    return dist;
}

Bipartite check (2-coloring)

pattern

Try to 2-color the graph with BFS/DFS; a conflict (neighbour with the same color) means it isn't bipartite. Appears as “can this be split into two groups” / “is this graph 2-colorable”.

cpp
bool isBipartite(vector<vector<int>>& adj, int n) {
    vector<int> color(n, -1);
    for (int s = 0; s < n; s++) {
        if (color[s] != -1) continue;
        queue<int> q; q.push(s); color[s] = 0;
        while (!q.empty()) {
            int u = q.front(); q.pop();
            for (int v : adj[u]) {
                if (color[v] == -1) { color[v] = color[u]^1; q.push(v); }
                else if (color[v] == color[u]) return false;
            }
        }
    }
    return true;
}
Graphs

Union-Find (DSU)

Union-Find with path compression + union by rank

★ high-freqtemplatewrite cold

Tracks connected components under merges in near-O(1) amortised (inverse-Ackermann). Path compression flattens lookups; union by rank keeps trees shallow. Memorise this whole struct — it appears constantly.

cpp
struct DSU {
    vector<int> parent, rnk;
    DSU(int n) : parent(n), rnk(n, 0) {
        iota(parent.begin(), parent.end(), 0);  // each its own root
    }
    int find(int x) {
        return parent[x] == x ? x
             : parent[x] = find(parent[x]);      // path compression
    }
    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;                // already connected
        if (rnk[a] < rnk[b]) swap(a, b);
        parent[b] = a;
        if (rnk[a] == rnk[b]) rnk[a]++;
        return true;
    }
};

What DSU solves

pattern

Reach for it whenever the problem is about grouping or connectivity under incremental unions: number of connected components, detect a cycle while adding edges (union returns false → the edge closes a cycle), redundant connection, accounts merge, and Kruskal's MST.

    if (!dsu.unite(u, v)) { /* u,v already connected → cycle */ }

Kruskal's MST

patternwrite cold

Minimum spanning tree: sort edges by weight, add each edge if it connects two different components (DSU union succeeds). Greedy + DSU, O(E log E).

long long kruskal(int n, vector<array<int,3>>& edges) { // {w, u, v}
    sort(edges.begin(), edges.end());
    DSU dsu(n);
    long long cost = 0; int used = 0;
    for (auto& [w, u, v] : edges)
        if (dsu.unite(u, v)) { cost += w; if (++used == n-1) break; }
    return cost;
}
Recursion

Recursion & Backtracking

The backtracking template

★ high-freqtemplate

Every backtracking problem is the same shape: choose an option, explore recursively, un-choose (undo). Prune branches that can't lead to a solution. Subsets, permutations, combinations, N-Queens, Sudoku, word search all instantiate this.

cpp
void backtrack(State& s, Result& res) {
    if (isComplete(s)) { res.push_back(s.current); return; }
    for (auto& choice : options(s)) {
        if (!valid(s, choice)) continue;  // prune
        apply(s, choice);                 // choose
        backtrack(s, res);                // explore
        undo(s, choice);                  // un-choose
    }
}

Subsets (power set)

★ high-freqwrite cold

At each index decide include / exclude. Recording cur at every call (not just leaves) collects all 2ⁿ subsets. Starting the loop at start avoids duplicate orderings.

cpp
void subsets(int start, vector<int>& nums,
             vector<int>& cur, vector<vector<int>>& res) {
    res.push_back(cur);                   // record every node
    for (int i = start; i < (int)nums.size(); i++) {
        cur.push_back(nums[i]);           // choose
        subsets(i + 1, nums, cur, res);   // explore (i+1: no reuse)
        cur.pop_back();                   // un-choose
    }
}

Permutations

★ high-freqwrite cold

Fix one position at a time by swapping each remaining element into place, recurse, swap back. O(n·n!) which is optimal since there are n! outputs.

cpp
void permute(vector<int>& nums, int k, vector<vector<int>>& res) {
    if (k == (int)nums.size()) { res.push_back(nums); return; }
    for (int i = k; i < (int)nums.size(); i++) {
        swap(nums[k], nums[i]);           // choose nums[i] for position k
        permute(nums, k + 1, res);
        swap(nums[k], nums[i]);           // restore
    }
}

Combinations & the include/exclude shape

patternwrite cold

Choose k of n, or all combinations summing to a target. The start index enforces increasing order so {1,2} and {2,1} aren't both produced. “Combination sum” reuses elements by recursing on i instead of i+1.

             vector<int>& cur, vector<vector<int>>& res) {

        cur.push_back(i);
        combine(i + 1, n, k, cur, res);
        cur.pop_back();

pattern

int solve(int r, int n, vector<bool>& col,
          vector<bool>& d1, vector<bool>& d2) {        // d1:r+c, d2:r-c+n

    int count = 0;
    for (int c = 0; c < n; c++) {

}
Search

Binary Search

The overflow-safe template

★ high-freqtemplatewrite cold

Binary search is easy to get subtly wrong (infinite loops, off-by-one, overflow). Use one canonical form. Compute mid as lo + (hi-lo)/2 — never (lo+hi)/2, which can overflow.

cpp
int binarySearch(vector<int>& a, int target) {
    int lo = 0, hi = a.size() - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;     // overflow-safe
        if (a[mid] == target) return mid;
        else if (a[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return -1;                            // not found
}

Lower / upper bound by hand

★ high-freqpatternwrite cold

int lowerBound(vector<int>& a, int x) {

Search in a rotated sorted array

★ high-freqpattern

One half of a rotated array is always sorted. Identify which, then decide whether the target lies inside it to discard the other half. O(log n) and a classic interview favourite.

cpp
int search(vector<int>& a, int target) {
    int lo = 0, hi = a.size() - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (a[mid] == target) return mid;
        if (a[lo] <= a[mid]) {                    // left half sorted
            if (a[lo] <= target && target < a[mid]) hi = mid - 1;
            else lo = mid + 1;
        } else {                                  // right half sorted
            if (a[mid] < target && target <= a[hi]) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return -1;
}

Binary search on the answer

★ high-freqpatternwrite cold

Sorting

Sorting

Know the sorts (and just call std::sort)

★ high-freq

In practice you call sort (O(n log n) introsort). But interviewers ask you to compare and sometimes implement. Key facts:

  • Merge sort — O(n log n) always, stable, O(n) extra space, the natural choice for linked lists & external sort.
  • Quick sort — O(n log n) average / O(n²) worst (bad pivots), in-place, not stable. Randomise the pivot.
  • Heap sort — O(n log n), in-place, not stable.
  • Counting / radix — O(n+k), beats the comparison lower bound when keys are small integers.

Merge sort (and the merge step)

patternwrite cold

Divide, sort halves, merge. The merge is the reusable piece — it's also how you count inversions. O(n log n), stable.

cpp
void mergeSort(vector<int>& a, int l, int r) {
    if (l >= r) return;
    int m = l + (r - l) / 2;
    mergeSort(a, l, m);
    mergeSort(a, m + 1, r);
    vector<int> tmp;
    int i = l, j = m + 1;
    while (i <= m && j <= r)
        tmp.push_back(a[i] <= a[j] ? a[i++] : a[j++]);
    while (i <= m) tmp.push_back(a[i++]);
    while (j <= r) tmp.push_back(a[j++]);
    for (int k = 0; k < (int)tmp.size(); k++) a[l + k] = tmp[k];
}

DP

Dynamic Programming

The DP framework

★ high-freq

Dynamic programming = recursion with overlapping subproblems cached. Four questions: state (what parameters define a subproblem?), transition (how do states combine?), base case, order (memoised top-down vs tabulated bottom-up). If you can write the recurrence, you can write the DP.

  • Top-down — recursion + a memo array; natural, only computes needed states.
  • Bottom-up — fill a table in dependency order; no recursion overhead, easier to space-optimise.

1D DP — climbing stairs / house robber

★ high-freqpatternwrite cold

The gateway problems. Each state depends on a couple of previous states, so you can drop the array to two rolling variables → O(1) space.

cpp
// House robber: max sum, no two adjacent
int rob(vector<int>& a) {
    int prev2 = 0, prev1 = 0;             // best up to i-2, i-1
    for (int x : a) {
        int cur = max(prev1, prev2 + x);  // skip x, or take x
        prev2 = prev1;
        prev1 = cur;
    }
    return prev1;
}

2D grid DP — unique paths / min path sum

★ high-freqpatternwrite cold

State = cell (r,c); each cell combines the cell above and to the left. The template for grid traversal counting and minimum-cost paths.

cpp
int minPathSum(vector<vector<int>>& g) {
    int R = g.size(), C = g[0].size();
    vector<vector<int>> dp(R, vector<int>(C, 0));
    for (int r = 0; r < R; r++)
        for (int c = 0; c < C; c++) {
            if (r == 0 && c == 0) dp[r][c] = g[r][c];
            else {
                int up   = r ? dp[r-1][c] : INT_MAX;
                int left = c ? dp[r][c-1] : INT_MAX;
                dp[r][c] = g[r][c] + min(up, left);
            }
        }
    return dp[R-1][C-1];
}

Longest common subsequence

★ high-freqpatternwrite cold

The archetype of string DP. dp[i][j] = LCS of the first i and first j chars. Matching chars extend the diagonal; otherwise take the better of dropping one char from either string. Edit distance, longest palindromic subsequence are variants.

cpp
int lcs(string& a, string& b) {
    int m = a.size(), n = b.size();
    vector<vector<int>> dp(m+1, vector<int>(n+1, 0));
    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            dp[i][j] = (a[i-1] == b[j-1])
                ? dp[i-1][j-1] + 1
                : max(dp[i-1][j], dp[i][j-1]);
    return dp[m][n];
}

Edit distance

patternwrite cold

Minimum insert/delete/replace to turn one string into another. Same grid as LCS; on a mismatch take 1 + the cheapest of the three operations.

cpp
int editDistance(string& a, string& b) {
    int m = a.size(), n = b.size();
    vector<vector<int>> dp(m+1, vector<int>(n+1));
    for (int i = 0; i <= m; i++) dp[i][0] = i;     // delete all
    for (int j = 0; j <= n; j++) dp[0][j] = j;     // insert all
    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            dp[i][j] = (a[i-1] == b[j-1])
                ? dp[i-1][j-1]
                : 1 + min({dp[i-1][j],     // delete
                           dp[i][j-1],     // insert
                           dp[i-1][j-1]}); // replace
    return dp[m][n];
}

0/1 Knapsack

★ high-freqpatternwrite cold

Each item taken at most once. The 1D rolling form iterates capacity backwards so each item is used once per row — forgetting that turns it into unbounded knapsack. Subset-sum and partition are special cases.

cpp
int knapsack(vector<int>& wt, vector<int>& val, int W) {
    vector<int> dp(W + 1, 0);
    for (int i = 0; i < (int)wt.size(); i++)
        for (int w = W; w >= wt[i]; w--)          // REVERSE → once each
            dp[w] = max(dp[w], dp[w - wt[i]] + val[i]);
    return dp[W];
}

Coin change (min coins / # of ways)

★ high-freqpatternwrite cold

Unbounded knapsack flavour. For minimum coins, iterate amounts forward and take the best over coins. For counting ways, loop coins outer / amount inner to avoid counting permutations.

cpp
int coinChange(vector<int>& coins, int amount) {
    vector<int> dp(amount + 1, amount + 1);       // "infinity"
    dp[0] = 0;
    for (int a = 1; a <= amount; a++)
        for (int c : coins)
            if (c <= a) dp[a] = min(dp[a], dp[a - c] + 1);
    return dp[amount] > amount ? -1 : dp[amount];
}

Longest increasing subsequence

★ high-freqpatternwrite cold

O(n²) DP is intuitive; the O(n log n) patience-sorting version is the impressive one — keep an array of smallest possible tails and binary-search each element's slot.

cpp
int lis(vector<int>& a) {                 // O(n log n)
    vector<int> tails;                    // tails[k] = smallest tail of len k+1
    for (int x : a) {
        auto it = lower_bound(tails.begin(), tails.end(), x);
        if (it == tails.end()) tails.push_back(x);  // extend
        else *it = x;                               // improve a tail
    }
    return tails.size();
}
Greedy

Greedy

When greedy actually works

★ high-freqgotcha

Greedy makes the locally optimal choice and never reconsiders. It only works when a problem has the greedy-choice property and optimal substructure — prove it (exchange argument) or at least sanity-check with examples, because a plausible greedy is often wrong (greedy coin change fails on {1,3,4} for 6). When unsure, DP is the safe fallback.

Interval scheduling (activity selection)

★ high-freqpatternwrite cold

Maximise non-overlapping intervals: sort by end time and greedily take each interval that starts after the last one taken. Sorting by end (not start) is the crucial, often-missed detail.

cpp
int maxNonOverlapping(vector<pair<int,int>>& iv) {  // (start, end)
    sort(iv.begin(), iv.end(), [](auto& a, auto& b){
        return a.second < b.second;                 // by end time
    });
    int count = 0, lastEnd = INT_MIN;
    for (auto& [s, e] : iv)
        if (s >= lastEnd) { count++; lastEnd = e; }  // take it
    return count;
}

Jump game / reach

patternwrite cold

Track the farthest index reachable as you sweep left to right; if you ever stand beyond it, you're stuck. O(n), no DP needed — a clean example of a provably-correct greedy.

cpp
bool canJump(vector<int>& a) {
    int farthest = 0;
    for (int i = 0; i < (int)a.size(); i++) {
        if (i > farthest) return false;          // unreachable
        farthest = max(farthest, i + a[i]);
    }
    return true;
}

Bit Manipulation

Bit operations you must know

★ high-freqpattern

Bit tricks show up in subset enumeration, state compression, and “do it without extra space” puzzles. The core moves:

  • Check bit i: (x >> i) & 1
  • Set bit i: x | (1 << i)
  • Clear bit i: x & ~(1 << i)
  • Toggle bit i: x ^ (1 << i)
  • Lowest set bit: x & -x
  • Clear lowest set bit: x & (x - 1)

XOR tricks (single number)

★ high-freqpatternwrite cold

x ^ x = 0 and x ^ 0 = x, so XORing every element cancels the pairs and leaves the unique one — O(n) time, O(1) space. XOR also swaps without a temp and finds a missing number.

cpp
int singleNumber(vector<int>& a) {
    int x = 0;
    for (int v : a) x ^= v;     // pairs cancel, lone survivor remains
    return x;
}

// missing number in 0..n
int missing(vector<int>& a) {
    int x = a.size();
    for (int i = 0; i < (int)a.size(); i++) x ^= i ^ a[i];
    return x;
}

Enumerate all subsets via bitmask

patternwrite cold

For n ≤ ~20, represent a subset as the bits of an integer 0..2ⁿ−1. The i-th bit means “element i is in the subset”. The basis of bitmask DP and brute-forcing small combinatorial problems.

cpp
int n = nums.size();
for (int mask = 0; mask < (1 << n); mask++) {
    vector<int> subset;
    for (int i = 0; i < n; i++)
        if (mask & (1 << i)) subset.push_back(nums[i]);
    // use subset...
}
Meta

Pattern Recognition

Signal → technique map

★ high-freqpattern

Interview problems are reskins of a small set of patterns. Train yourself to map the phrasing/constraints to the tool. This table is the fastest way to find the approach under pressure.

  • Sorted array, find pair/triplet → two pointers (or binary search)
  • “Contiguous subarray / substring” with a constraint → sliding window
  • Range-sum queries → prefix sums
  • “Top / k-th largest or smallest” → heap (or quickselect)
  • “Next greater / smaller”, histogram → monotonic stack
  • “Max in every window of size k” → monotonic deque
  • Pairing / complement / dedupe / seen-before → hashmap / hashset
  • Search space is sorted or answer is monotonic → binary search (on array or on answer)
  • “All combinations / permutations / partitions” → backtracking
  • Shortest path, unweighted → BFS; weighted, non-negative → Dijkstra
  • Dependencies / ordering / build order → topological sort
  • Connectivity / grouping / merge accounts → union-find (DSU)
  • Grid / matrix traversal, islands, flood fill → BFS / DFS on cells
  • “Min/max cost”, “# of ways”, overlapping subproblems → dynamic programming
  • “Maximise/minimise with a locally-optimal choice” → greedy (then verify)
  • Tiny n (≤ ~20), subsets of items → bitmask / brute force
  • Tree — “path / ancestor / level” → DFS recursion / BFS by levels

If you're stuck, run this checklist

★ high-freq

When nothing jumps out, work through escalating ideas out loud — interviewers reward structured exploration even before you land the optimal answer.

  • Can sorting the input reveal structure or enable two pointers / binary search?
  • What should I store to avoid recomputation? (hashmap, prefix sums, DP table)
  • Is the brute force exploring overlapping subproblems? → memoise → DP.
  • Can I trade space for time, or do it in-place to save space?
  • Is the answer monotonic in some value? → binary search on the answer.
  • Reframe as a graph: nodes, edges, then BFS/DFS/DSU/topo.
Pitfalls

C++ Gotchas

Integer overflow (again — it's that common)

★ high-freqgotcha

Sums of large arrays, products, mid = (lo+hi)/2, and a*b % m all overflow int silently. Default to long long for sums/products that can exceed ~2×109, and cast before multiplying. This is the #1 cause of “passes samples, fails hidden tests”.

Iterator / reference invalidation

★ high-freqgotcha

Growing a vector (push_back) can reallocate and invalidate every pointer, reference, and iterator into it. So a reference to an element taken before a push may dangle afterwards. Don't hold references across modifications; index instead, or reserve up front.

cpp
vector<int> v = {1, 2, 3};
int& ref = v[0];
v.push_back(4);        // may reallocate → ref now dangles!
// ref = 10;           // undefined behaviour

for (int x : v) if (x == 2) v.push_back(99); // also UB: modifying while ranging

Comparator must be a strict weak ordering

gotcha

A sort/set comparator must return false for equal elements (use <, never <=). A non-strict comparator is undefined behaviour and can crash at runtime on large inputs — a nasty, hard-to-reproduce bug.

cpp
sort(v.begin(), v.end(), [](int a, int b){ return a <= b; }); // WRONG (UB)
sort(v.begin(), v.end(), [](int a, int b){ return a <  b; }); // correct

map[key] inserts; multiset erase removes all

gotcha

Accessing m[k] on a map default-constructs and inserts k if absent — so a read can silently grow the map (and break a loop over it). Use .count()/.find() to test membership. And multiset::erase(value) erases every copy; erase one with ms.erase(ms.find(value)).

cpp
if (m.count(k)) use(m[k]);     // safe read
// if (m[k] != 0) ...          // inserts k as a side effect!

ms.erase(ms.find(x));          // remove ONE x, not all

Common TLE causes

★ high-freqgotcha

If a correct solution times out, the usual suspects are:

  • Slow I/O — forgot sync_with_stdio(false) / cin.tie(nullptr), or used endl in a tight loop.
  • Passing big containers by value into functions or recursion.
  • Building strings with s = s + c in a loop (O(n²)); use +=.
  • Wrong complexity for the constraints — re-read the bounds and re-derive the target.
  • unordered_map worst-case collisions on adversarial input — try map or a custom hash.

Floating point & edge inputs

gotcha

Never compare floats with == — use an epsilon (abs(a-b) < 1e-9). And always sanity-check the boundary inputs interviewers probe: empty container, single element, all-equal elements, already-sorted/reverse-sorted, negatives, and integer limits.

JavaScript Core

Closures

★ high-freq

A function bundled with references to its surrounding (lexical) scope. The inner function keeps access to outer variables even after the outer function returns. Powers data privacy, memoization, and function factories.

javascript
function counter() {
  let count = 0;                 // private state
  return () => ++count;          // closes over `count`
}
const next = counter();
next(); // 1
next(); // 2

The classic loop + closure bug

★ high-freqgotcha

var is function-scoped, so every callback shares one binding and reads the final value. let creates a fresh binding per iteration. This is the single most asked closure question.

javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 3, 3, 3
}

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 0, 1, 2
}

// Pre-let fix: IIFE captures the current value
for (var i = 0; i < 3; i++) {
  ((j) => setTimeout(() => console.log(j), 0))(i);
}

var / let / const, hoisting & the TDZ

★ high-freqgotcha

Declarations are hoisted; var initialises to undefined, while let/const sit in the Temporal Dead Zone until their line runs and throw if accessed early. var is function-scoped; let/const are block-scoped. const binds a constant reference, not an immutable value.

javascript
console.log(a); // undefined (var hoisted, initialised)
var a = 1;

console.log(b); // ReferenceError (TDZ)
let b = 2;

const obj = { x: 1 };
obj.x = 2;       // OK — object is mutable
// obj = {};     // TypeError — binding is constant

this binding

★ high-freqgotcha

Four rules, checked in priority order: new > explicit (call/apply/bind) > implicit (method call obj.fn()) > default (undefined in strict mode, global otherwise). Arrow functions ignore all of this and capture this lexically from where they are defined.

javascript
const obj = {
  name: 'A',
  regular() { return this.name; },  // 'A'  (implicit)
  arrow: () => this.name,           // outer this (not obj)
};

function f() { return this; }
f.call(obj);          // obj   (explicit)
f.bind(obj)();        // obj
new f();              // fresh object

call / apply / bind

call invokes now with args listed; apply invokes now with args as an array; bind returns a new function with this permanently set.

javascript
fn.call(ctx, a, b);
fn.apply(ctx, [a, b]);
const bound = fn.bind(ctx, a); // partial application + fixed this

// Borrow array methods on array-likes
Math.max.apply(null, [1, 5, 2]);          // 5
Array.prototype.slice.call(arguments);    // real array

Prototypes & prototypal inheritance

★ high-freq

Every object has a hidden link ([[Prototype]]) to another object. Property lookups walk this chain until found or null. class is syntactic sugar over this. __proto__ is the link on instances; prototype is the object new instances will link to.

javascript
function Animal(name) { this.name = name; }
Animal.prototype.speak = function () {
  return this.name + ' makes a sound';
};

const dog = new Animal('Rex');
dog.speak();                       // walks proto chain
Object.getPrototypeOf(dog) === Animal.prototype; // true

// Equivalent with class
class Cat extends Animal {
  speak() { return this.name + ' meows'; }
}

What new actually does

implement

Creates a fresh object, links its prototype to the constructor's prototype, runs the constructor with this bound to it, and returns that object unless the constructor returns its own object.

javascript
function myNew(Ctor, ...args) {
  const obj = Object.create(Ctor.prototype);
  const ret = Ctor.apply(obj, args);
  return ret instanceof Object ? ret : obj;
}

== vs === and coercion

★ high-freqgotcha

=== compares value and type with no coercion. == coerces and has surprising edges — use === everywhere except the idiom x == null (true for null and undefined). Falsy values: false, 0, -0, 0n, '', null, undefined, NaN.

javascript
0 == '';          // true   (both coerce to 0)
0 == '0';         // true
'' == '0';        // false
null == undefined;// true
NaN === NaN;      // false  -> use Number.isNaN()
[] == ![];        // true   (classic trick)
typeof null;      // 'object' (historic bug)

Pass by value vs reference

Primitives are copied by value; objects/arrays/functions are passed by a copy of their reference. Reassigning a parameter never affects the caller; mutating the referenced object does.

javascript
function mutate(o) { o.x = 2; }     // visible to caller
function reassign(o) { o = { x: 9 }; } // not visible
const a = { x: 1 };
mutate(a);   // a.x === 2
reassign(a); // a.x still 2

Shallow vs deep copy

★ high-freq

Spread and Object.assign copy one level — nested objects are still shared. structuredClone deep-copies (handles cycles, Dates, Maps). JSON.parse(JSON.stringify(x)) works for plain data only — it drops functions, undefined, and breaks on Date/cycles.

javascript
const shallow = { ...obj };           // 1 level
Object.assign({}, obj);               // 1 level
const deep = structuredClone(obj);    // full deep copy
JSON.parse(JSON.stringify(obj));      // plain JSON only

Destructuring, spread & rest

Spread expands; rest collects. Destructuring pulls values out with defaults and renaming.

javascript
const [first, ...others] = [1, 2, 3];      // 1, [2,3]
const { a, b = 10, c: renamed } = obj;
const merged = { ...base, ...override };   // later wins
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }

Optional chaining & nullish coalescing

gotcha

?. short-circuits to undefined instead of throwing on null/undefined. ?? falls back only for null/undefined (unlike ||, which also triggers on 0, '', false).

javascript
user?.address?.city;          // undefined if any link is null
config.count ?? 10;           // 10 only if null/undefined
0 || 5;   // 5   (probably a bug)
0 ?? 5;   // 0   (usually what you want)

Modules: ESM vs CommonJS

ESM (import/export) is static, tree-shakeable, and async; the browser and modern Node use it. CommonJS (require/module.exports) is dynamic and synchronous — the legacy Node system.

javascript
// ESM
import { foo } from './a.js';
export const bar = 1;
export default fn;

// CommonJS
const { foo } = require('./a');
module.exports = { bar };

Generators & iterators

A generator (function*) pauses at yield and resumes on .next(), producing lazy sequences. Any object implementing Symbol.iterator works with for...of and spread.

javascript
function* ids() {
  let i = 1;
  while (true) yield i++;   // infinite, lazy
}
const gen = ids();
gen.next().value; // 1
gen.next().value; // 2

Map / Set / WeakMap

Map keeps insertion order and allows any key type (objects included); Set stores unique values. WeakMap/WeakSet hold keys weakly so entries are garbage-collected — good for private data and caches keyed by objects.

javascript
const seen = new Set([1, 1, 2]);   // {1, 2}
[...new Set(arr)];                 // dedupe an array

const cache = new WeakMap();       // entries GC'd with the key
cache.set(domNode, data);

Higher-order functions: map / filter / reduce

Functions that take or return functions. map transforms, filter selects, reduce folds to a single value. Prefer them over manual loops for clarity.

javascript
const nums = [1, 2, 3, 4];
nums.map((n) => n * 2);                 // [2,4,6,8]
nums.filter((n) => n % 2 === 0);        // [2,4]
nums.reduce((acc, n) => acc + n, 0);    // 10

Async JS & Event Loop

The event loop model

★ high-freq

JS runs on one thread with a call stack. Async work (timers, fetch, DOM events) is handed to the host (browser/Node) and its callback is queued. When the stack is empty, the loop drains all microtasks (Promise callbacks, queueMicrotask) first, then one macrotask (setTimeout, I/O, UI events), then repeats.

javascript
// Mental model
// stack -> (empty?) -> drain ALL microtasks -> 1 macrotask -> repeat

Microtask vs macrotask ordering

★ high-freqgotcha

The canonical output question. Synchronous code runs first, then microtasks (Promises), then macrotasks (timers) — regardless of a 0ms delay.

javascript
console.log('1');
setTimeout(() => console.log('2'), 0);   // macrotask
Promise.resolve().then(() => console.log('3')); // microtask
console.log('4');

// Output: 1, 4, 3, 2

Promises & chaining

A Promise is pending, then fulfilled or rejected. .then returns a new promise so calls chain; a thrown error or rejection skips to the nearest .catch. .finally always runs.

javascript
fetchUser()
  .then((u) => fetchPosts(u.id))   // return a promise to chain
  .then((posts) => render(posts))
  .catch((err) => showError(err)) // catches the whole chain
  .finally(() => stopSpinner());

Promise.all / allSettled / race / any

★ high-freq

Know all four cold. all rejects fast on first failure; allSettled never rejects and reports each outcome; race settles with the first to settle (resolve or reject); any resolves with the first success and rejects only if all fail.

javascript
Promise.all([a, b]);        // [ra, rb] | rejects on first error
Promise.allSettled([a, b]); // [{status,value|reason}, ...]
Promise.race([a, b]);       // first to settle (win or fail)
Promise.any([a, b]);        // first to succeed

async / await

Sugar over promises: await pauses the async function until the promise settles, and try/catch handles rejection like sync errors. An async function always returns a promise.

javascript
async function load() {
  try {
    const user = await fetchUser();
    const posts = await fetchPosts(user.id);
    return posts;
  } catch (err) {
    handle(err);
  }
}

Sequential vs parallel awaits

★ high-freqperf

A very common performance gotcha. Awaiting independent calls one-by-one runs them in series. Kick them off together and await with Promise.all.

javascript
// Slow: total = a + b  (sequential)
const x = await getA();
const y = await getB();

// Fast: total = max(a, b)  (parallel)
const [x, y] = await Promise.all([getA(), getB()]);

Cancelling async work: AbortController

pattern

For animation, use requestAnimationFrame: it runs right before the next paint, syncs to the display refresh rate, and pauses in background tabs. setTimeout drifts and can paint mid-frame.

Box model & box-sizing

★ high-freq

Every box = content + padding + border + margin. By default width sizes only the content. Set box-sizing: border-box so width includes padding and border — the sane default applied globally in most resets.

★ high-freq

★ high-freq

★ high-freq

Transitions, transforms & animation perf

perf

Reflow vs repaint

perf

Reflow/layout recalculates geometry (changing size, position, adding nodes) and is expensive. Repaint redraws pixels without geometry (color, visibility). Composite only moves existing layers (transform/opacity) and is cheapest. Batch DOM reads then writes to avoid layout thrashing.

javascript

// Better: read all first, then write all

★ high-freqgotcha

useEffect: deps & cleanup

★ high-freqgotcha

Runs after render for side effects (subscriptions, fetches, timers). The dependency array controls when it re-runs; [] = once on mount. Return a cleanup function to undo work before the next run or on unmount. Most bugs come from missing or wrong deps and stale closures.

jsx

  return () => clearInterval(id);   // cleanup
}, []);                              // run once

}, [userId]);                        // re-run when userId changes

HTTP methods, safety & idempotency

★ high-freq

★ high-freqsecurity

Attacker injects script that runs in your origin and steals data/sessions. Types: stored, reflected, DOM-based. function deepClone(val, seen = new WeakMap()) {
  if (val === null || typeof val !== 'object') return val;
  if (seen.has(val)) return seen.get(val);
  if (val instanceof Date) return new Date(val);
  const copy = Array.isArray(val) ? [] : {};
  seen.set(val, copy);
  for (const key of Reflect.ownKeys(val)) {
    copy[key] = deepClone(val[key], seen);

  return copy;

LRU cache

★ high-freqimplement

Evict the least-recently-used entry at capacity. A Map preserves insertion order, so the first key is the LRU.

javascript
class LRUCache {
  constructor(capacity) { this.cap = capacity; this.map = new Map(); }
  get(key) {
    if (!this.map.has(key)) return -1;
    const v = this.map.get(key);
    this.map.delete(key); this.map.set(key, v);   // refresh recency
    return v;
  }
  put(key, v) {
    if (this.map.has(key)) this.map.delete(key);
    else if (this.map.size >= this.cap)
      this.map.delete(this.map.keys().next().value); // evict LRU
    this.map.set(key, v);
  }
}

OSI is a 7-layer reference model (Physical, Data Link, Network, Transport, Session, Presentation, Application); TCP/IP is the practical 4-layer model the internet actually runs on (Network Access, Internet, Transport, Application). Interviewers want you to name the layers bottom-to-top and give a one-line role for each, then map a protocol to its layer.

  • Application — HTTP, DNS, SMTP, FTP
  • Transport — TCP, UDP (ports, end-to-end delivery)
  • Network/Internet — IP, ICMP (routing, addressing)
  • Data Link / Physical — MAC, Ethernet, the wire

Switch vs router vs hub

devices

A hub blindly repeats traffic to all ports (Layer 1). A switch forwards frames only to the right device using MAC addresses within a LAN (Layer 2). A router connects different networks and forwards packets using IP addresses, choosing the best path (Layer 3).

Encapsulation

models

As data moves down the stack each layer wraps it with its own header (and the data link layer adds a trailer): segment → packet → frame → bits. The receiver reverses this (de-encapsulation). It's why a single HTTP request ends up with TCP, IP, and Ethernet headers stacked around it.

Transport

TCP, UDP & Ports

TCP vs UDP

★ high-freqtransport

The most-asked networking question. TCP is connection-oriented and reliable — it sets up a connection, guarantees ordered delivery, retransmits lost data, and does flow/congestion control (web, email, file transfer). UDP is connectionless and fast with no delivery or ordering guarantees and minimal overhead (live video, gaming, VoIP, DNS).

The follow-up is always why real-time apps prefer UDP: waiting for TCP to retransmit a lost packet causes lag, and a slightly dropped frame matters less than latency.

TCP 3-way handshake

★ high-freqtransport

How a TCP connection is established: SYN (client → server, “let's talk, here's my sequence number”), SYN-ACK (server → client, acknowledges and sends its own sequence number), ACK (client → server, acknowledges). After these three messages both sides are synchronised and ready to send data.

Connection teardown (4-way)

transport

Closing is four steps because each side closes its half independently: FIN → ACK → FIN → ACK. The initiator then waits in TIME_WAIT for any straggling packets before fully releasing the connection.

Flow control vs congestion control

gotchatransport

Both throttle the sender but for different reasons. Flow control stops a fast sender from overwhelming a slow receiver (sliding window / advertised receive window). Congestion control stops senders from overwhelming the network (slow start, AIMD, congestion window). Mixing these two up is a common slip.

How does TCP guarantee reliability?

transport

Sequence numbers (order + detect loss), acknowledgements (confirm receipt), retransmission on timeout or duplicate ACKs, checksums (detect corruption), and windowing for flow control. UDP has only a basic checksum and no recovery.

Ports & sockets

transport

A port is a 16-bit number (0–65535) identifying a process on a host — well-known ports include 80 (HTTP), 443 (HTTPS), 22 (SSH), 53 (DNS). A socket is the endpoint of a connection, identified by the (IP, port) pair; a full TCP connection is the 4-tuple (src IP, src port, dst IP, dst port). Gotcha: TCP:80 and UDP:80 are independent, so two services can use the same port number on different protocols.

Web

HTTP, HTTPS & TLS

HTTP methods & idempotency

★ high-freqhttp

GET (read, safe), POST (create, not idempotent), PUT (replace, idempotent), PATCH (partial update), DELETE (idempotent). Idempotent = repeating the request has the same effect as doing it once; this matters for safe retries. Expect a follow-up on which methods are safe vs idempotent.

HTTP status codes

★ high-freqhttp

Know the families and the common ones: 2xx success (200 OK, 201 Created, 204 No Content), 3xx redirect (301 permanent, 302 found, 304 Not Modified), 4xx client error (400, 401 unauthenticated, 403 forbidden, 404, 429 Too Many Requests), 5xx server error (500, 502, 503). 401 vs 403 (“who are you” vs “you can't”) is a favourite.

HTTP/1.1 vs HTTP/2 vs HTTP/3

http

HTTP/1.1 sends one request per connection at a time (head-of-line blocking). HTTP/2 adds multiplexing (many streams over one TCP connection), header compression, and server push. HTTP/3 runs over QUIC (UDP), removing TCP-level head-of-line blocking and cutting handshake round-trips.

HTTPS & the TLS handshake

★ high-freqhttps

HTTPS = HTTP over TLS (port 443). The handshake: client sends supported TLS versions/ciphers → server returns its certificate (containing its public key, signed by a CA) → client verifies the certificate and they agree on a shared session key → all further traffic is symmetrically encrypted. It gives confidentiality, integrity, and server authentication. TLS 1.3 reduces this to fewer round-trips.

Cookies vs sessions vs tokens

http

Cookies are small key-values the browser stores and sends back automatically. Sessions keep state server-side (cookie holds only a session ID) — stateful, instantly revocable. Tokens (JWT) are self-contained and signed — stateless, scale across services, but can't be revoked before expiry. HTTP itself is stateless; these layer state on top.

Why is HTTP called stateless?

http

Each request is independent — the server keeps no memory of previous requests on the connection. State (login, cart) is reconstructed each time via cookies, sessions, or tokens. This is what makes web servers easy to scale horizontally.

WebSockets vs HTTP polling

http

HTTP is request/response and one-directional per exchange. WebSockets upgrade a single connection to a persistent, full-duplex channel so the server can push to the client in real time (chat, live feeds) — far cheaper than polling or long-polling.

Addressing

DNS, IP & Addressing

DNS resolution — step by step

★ high-freqdns

DNS turns a domain into an IP. Resolution walks a hierarchy: browser/OS cache → recursive resolver → root server → TLD server (.com) → authoritative server, which returns the IP, cached along the way (TTL). Records: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), NS (nameserver). Runs mostly over UDP:53.

Public vs private IP, and NAT

addressing

Private IP ranges (10.x, 172.16–31.x, 192.168.x) are used inside local networks and aren't routable on the internet. NAT (Network Address Translation) lets many private devices share one public IP by rewriting addresses/ports at the router — the main reason we haven't run out of IPv4 yet.

IPv4 vs IPv6

addressing

IPv4 is 32-bit (~4.3 billion addresses, dotted decimal); IPv6 is 128-bit (effectively unlimited, hex, written like 2001:db8::1). IPv6 also simplifies headers and removes the need for NAT.

MAC vs IP address (and ARP)

gotchaaddressing

A MAC address is the permanent physical address of a NIC, used for delivery within a local network (Layer 2). An IP address is logical and used to route across networks (Layer 3). ARP (Address Resolution Protocol) maps a known IP to its MAC on the local network so a frame can actually be delivered.

Subnetting & CIDR (the idea)

addressing

Subnetting splits a network into smaller ones; the subnet mask says which bits are the network vs host portion. CIDR notation like 192.168.1.0/24 means the first 24 bits are the network, leaving 8 bits (256 addresses) for hosts. You usually only need the concept and to compute host counts, not heavy bit math.

DHCP

addressing

DHCP automatically assigns IP addresses and config (gateway, DNS) to devices joining a network, via the DORA exchange: Discover → Offer → Request → Acknowledge. Saves manual IP assignment.

How the web works

How the Web Works

What happens when you type google.com and hit Enter?

★ high-freqthe-flow

The flagship networking question — they want the full end-to-end chain, not “it goes to the server.”

  • DNS lookup — resolve google.com to an IP (cache → resolver → root → TLD → authoritative).
  • TCP handshake — SYN / SYN-ACK / ACK to the server (often via a load balancer).
  • TLS handshake — negotiate keys for HTTPS.
  • HTTP request — browser sends GET; server (behind LB/CDN) responds with HTML.
  • Render — browser parses HTML, fetches CSS/JS/images, builds the DOM, paints.

Strong answers mention ARP/default gateway and routing between the layers.

Load balancer — what and why

★ high-freqthe-flow

Distributes incoming traffic across multiple servers for availability and scale, and removes a single point of failure. Common algorithms: round-robin, least-connections, IP-hash. Operates at L4 (transport) or L7 (application, can route by URL/header). The reason a stateless web tier scales horizontally.

CDN

the-flow

A Content Delivery Network caches static assets (images, JS, CSS, video) on edge servers geographically close to users, cutting latency and origin load. Edge picks the nearest/fastest node; cache freshness is handled with TTLs and invalidation.

Forward proxy vs reverse proxy

gotchathe-flow

A forward proxy sits in front of clients and makes requests on their behalf (corporate filtering, anonymity). A reverse proxy sits in front of servers and fronts them to the world (load balancing, TLS termination, caching) — e.g. Nginx.

Latency vs bandwidth

the-flow

Latency is the delay for one packet to travel (ms); bandwidth is how much data per second the link carries. A fat pipe with high latency still feels slow for chatty requests — they're independent. Throughput is the actual achieved rate, bounded by both.

Foundations

Fundamentals & Keys

DBMS vs RDBMS

basics

A DBMS stores and manages data; an RDBMS is a DBMS that stores data in related tables (rows/columns) with relationships enforced via keys and queried with SQL (MySQL, PostgreSQL, Oracle). RDBMS adds referential integrity and ACID guarantees on top of basic storage.

Types of keys

★ high-freqbasics

Know all of these — a standard rapid-fire:

  • Super key — any set of columns that uniquely identifies a row.
  • Candidate key — a minimal super key (no redundant column).
  • Primary key — the chosen candidate key; unique, not NULL.
  • Foreign key — a column referencing another table's primary key (referential integrity).
  • Composite key — a key made of two or more columns.
  • Unique key — like a primary key but allows one NULL.

Primary key vs unique key

gotchabasics

Both enforce uniqueness, but a table has exactly one primary key which cannot be NULL, while it can have many unique keys and a unique column may hold a NULL. Primary key is the row's identity; unique just forbids duplicates.

DDL, DML, DCL, TCL

basics

SQL command families: DDL defines schema (CREATE, ALTER, DROP, TRUNCATE), DML manipulates data (SELECT, INSERT, UPDATE, DELETE), DCL controls access (GRANT, REVOKE), TCL manages transactions (COMMIT, ROLLBACK, SAVEPOINT).

ER model & schema vs instance

basics

An ER (entity-relationship) model is a high-level diagram of entities, their attributes, and relationships, used to design the database before tables exist. The schema is the structure/blueprint (fixed); the instance/state is the actual data at a moment (changes constantly).

Design

Normalization

What is normalization (and why)?

★ high-freqdesign

Organising tables to minimise redundancy and avoid update, insert, and delete anomalies, by splitting large tables into smaller related ones based on functional dependencies. Trade-off: more joins. Almost always tested with “explain the normal forms.”

1NF → 2NF → 3NF → BCNF

★ high-freqdesign

The progression interviewers want you to recite:

  • 1NF — atomic values, no repeating groups/arrays in a column.
  • 2NF — 1NF + no partial dependency (non-key columns depend on the whole composite key).
  • 3NF — 2NF + no transitive dependency (non-key columns depend only on the key, not other non-key columns).
  • BCNF — a stricter 3NF; every determinant is a candidate key (handles edge cases with overlapping composite keys).

Normalization vs denormalization

design

Normalization reduces redundancy (better integrity, more joins, good for write-heavy OLTP). Denormalization deliberately adds redundancy to cut joins and speed reads (good for read-heavy / analytics / OLAP). It's a read-speed vs write-cost / storage trade-off, not a “right answer.”

Database anomalies

design

Problems redundancy causes: update anomaly (must change the same fact in many rows), insertion anomaly (can't add data without unrelated data), deletion anomaly (deleting a row loses unrelated facts). Normalization exists to remove these.

Querying

SQL Queries

Types of JOINs

★ high-freqsql

INNER — only matching rows in both tables. LEFT — all left rows + matches (NULLs where none). RIGHT — mirror of left. FULL OUTER — all rows from both. CROSS — Cartesian product. SELF — a table joined to itself (e.g. employee→manager). Be ready to write one.

sql
SELECT e.name, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

WHERE vs HAVING

★ high-freqgotchasql

WHERE filters individual rows before grouping and can't use aggregates. HAVING filters after GROUP BY and can use aggregates like COUNT(). Mixing them up is a classic mistake.

sql
SELECT dept, COUNT(*) AS n
FROM employees
WHERE active = 1            -- row filter (pre-group)
GROUP BY dept
HAVING COUNT(*) > 5;        -- group filter (post-group)

DELETE vs TRUNCATE vs DROP

★ high-freqsql

DELETE (DML) removes selected rows, can have a WHERE, is logged and rollback-able, fires triggers. TRUNCATE (DDL) wipes all rows fast, no WHERE, resets identity, minimal logging. DROP (DDL) removes the whole table structure. A very common one-liner question.

UNION vs UNION ALL

sql

Both stack result sets with matching columns. UNION removes duplicates (so it sorts/dedupes — slower); UNION ALL keeps everything including duplicates (faster). Use UNION ALL unless you actually need dedup.

Find the Nth highest salary

★ high-freqsql

A perennial live-SQL ask. The portable approach uses a window function (or a correlated subquery / LIMIT n-1,1 in MySQL).

sql
-- 2nd highest salary
SELECT DISTINCT salary
FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
) t
WHERE rnk = 2;

How is NULL different from 0 or ''?

gotchasql

NULL means unknown / not applicable — it is not zero and not an empty string. Comparisons with NULL yield UNKNOWN (so x = NULL is never true; use IS NULL), and most aggregates skip NULLs. A frequent trap.

Concurrency

Transactions & Concurrency

ACID properties

★ high-freqtransactions

The guarantees that make transactions safe:

  • Atomicity — all-or-nothing; partial failure rolls back.
  • Consistency — moves the DB from one valid state to another (constraints hold).
  • Isolation — concurrent transactions don't interfere with each other.
  • Durability — once committed, survives crashes.

Isolation levels & read phenomena

★ high-freqtransactions

Higher isolation prevents more anomalies but reduces concurrency:

  • Read Uncommitted — allows dirty reads (reading uncommitted data).
  • Read Committed — no dirty reads; allows non-repeatable reads (default in many DBs).
  • Repeatable Read — same row reads stay stable; can allow phantom reads.
  • Serializable — strongest, as if transactions ran one-by-one; slowest.

Dirty / non-repeatable / phantom reads

gotchatransactions

Dirty read — you read data another transaction hasn't committed (it may roll back). Non-repeatable read — you read a row twice and get different values because another txn updated it. Phantom read — you run the same query twice and new rows appear because another txn inserted them.

Locks & deadlocks

transactions

A shared (read) lock lets many readers; an exclusive (write) lock blocks everyone else. A deadlock is two transactions each holding a lock the other needs. Prevent/handle with consistent lock ordering, short transactions, timeouts, and the DB's deadlock detector (it aborts a victim).

Optimistic vs pessimistic concurrency

transactions

Pessimistic locks data up front assuming conflicts are likely (good for high contention). Optimistic assumes conflicts are rare — it checks a version/timestamp at commit and retries if something changed (good for low contention, better throughput).

Performance

Indexing & Optimization

What is an index, and the trade-off?

★ high-freqindexing

A data structure (usually a B+ tree) that lets the DB find rows without a full table scan — turning O(n) lookups into O(log n). The trade-off: indexes speed up reads but slow down writes (every insert/update/delete must maintain them) and use storage. So index selectively, on columns used in WHERE/JOIN/ORDER BY.

Clustered vs non-clustered index

★ high-freqindexing

A clustered index defines the physical order of rows on disk — one per table (often the primary key). A non-clustered index is a separate structure with pointers to the rows — you can have many. Range scans are faster on the clustered key.

Composite & covering indexes

indexing

A composite index spans multiple columns; order matters — it helps queries that filter on a left-most prefix of the columns. A covering index contains all columns a query needs, so the DB answers from the index alone without touching the table.

When does an index hurt?

gotchaindexing

On small tables (scan is cheaper), on low-cardinality columns (e.g. boolean), and on write-heavy tables where maintenance cost outweighs read gains. Too many indexes bloat storage and slow writes.

Query optimization & EXPLAIN

★ high-freqindexing

Use EXPLAIN to read the execution plan and spot full table scans, missing indexes, or bad join order. Fixes: add the right index, rewrite the query, avoid SELECT *, filter early, and watch the N+1 problem (one query per row in a loop — batch with a join or IN instead).

Scale

SQL vs NoSQL & Scale

SQL vs NoSQL — when to use which

★ high-freqscale

SQL: fixed schema, strong consistency (ACID), powerful joins, scales vertically — great for transactions and complex relationships (banking, orders). NoSQL: flexible schema, scales horizontally, optimised for high read/write throughput, follows BASE/CAP — great for huge or unstructured data (feeds, logs, catalogs). Don't say one is “better” — frame it as a trade-off.

Types of NoSQL databases

scale

Key-value (Redis, DynamoDB) — fast lookups/cache. Document (MongoDB) — JSON-like, flexible. Column-family (Cassandra) — wide rows, write-heavy at scale. Graph (Neo4j) — relationship-heavy data (social graphs, recommendations).

CAP theorem

★ high-freqscale

A distributed datastore can guarantee at most two of Consistency, Availability, and Partition tolerance. Since network partitions are unavoidable, the real choice under a partition is C vs A: CP systems reject requests to stay consistent; AP systems stay up and reconcile later (eventual consistency).

ACID vs BASE

scale

ACID (relational) prioritises strong consistency. BASE (many NoSQL) — Basically Available, Soft state, Eventual consistency — trades immediate consistency for availability and scale. It's the CAP trade-off expressed as a design philosophy.

Sharding & partitioning

scale

Horizontal partitioning (sharding) splits rows across nodes by a shard key (user ID, region) to scale writes and storage. Vertical partitioning splits columns. Hard parts: choosing a shard key that avoids hot spots, and cross-shard queries/joins. Often paired with consistent hashing and replication.

OLTP vs OLAP

scale

OLTP — many small, fast transactions, highly normalised (the live app database). OLAP — few large analytical queries over big datasets, often denormalised (data warehouse, star/snowflake schema). Different shapes for different workloads.

Vulnerabilities

Common Web Vulnerabilities

SQL Injection — and how to prevent it

★ high-freqowasp

Injecting SQL through an input so it changes the query — e.g. ' OR 1=1-- in a login field to bypass auth or dump data. Root cause: mixing code with untrusted input. Prevention: parameterized queries / prepared statements (the #1 fix), ORMs, allow-list input validation, and least-privilege DB accounts. Manual escaping is not reliable.

sql
# BAD  - string concatenation is injectable
db.execute("SELECT * FROM users WHERE name = '" + name + "'")

# GOOD - parameterized query
db.execute("SELECT * FROM users WHERE name = ?", (name,))

Cross-Site Scripting (XSS)

★ high-freqowasp

Injecting malicious JavaScript that runs in other users' browsers — e.g. a comment containing <script>...</script> that steals cookies. Three types: stored (saved on the server), reflected (echoed from the request), DOM-based (client-side JS writes untrusted input to the DOM). Prevention: output-encode/escape user content, a Content-Security-Policy header, and HttpOnly cookies so JS can't read them.

Cross-Site Request Forgery (CSRF)

★ high-freqgotchaowasp

Tricks a logged-in user's browser into making an unwanted state-changing request (the browser auto-sends cookies). Prevention: anti-CSRF tokens (unique, unpredictable, per session), the SameSite cookie attribute (Lax/Strict), and checking the Origin/Referer header. Note: XSS targets other users via injected script; CSRF abuses an existing authenticated session — don't conflate them.

SSRF & IDOR

owasp

SSRF (Server-Side Request Forgery) tricks the server into making requests to internal resources it shouldn't reach (cloud metadata, internal services). IDOR (Insecure Direct Object Reference) is accessing another user's data by changing a predictable ID in the URL (/orders/124 → /orders/125) — fix with server-side authorization checks on every object.

OWASP Top 10

owasp

The industry-standard list of the most critical web app risks — currently led by Broken Access Control, plus Injection, Cryptographic Failures, Security Misconfiguration, Vulnerable Components, Identification/Auth failures, and more. Interviewers expect you to know it exists and name a few with mitigations.

Identity

Authentication & Sessions

Authentication vs authorization

★ high-freqauth

Authentication = who are you (verify identity: password, MFA, OAuth login). Authorization = what are you allowed to do (permissions, roles, scopes). Authentication always comes first. A one-line favourite.

Session vs JWT (token) auth

★ high-freqauth

Session: server stores session state, client holds a session-ID cookie — stateful, instantly revocable, needs a shared store across servers. JWT: signed self-contained token the client sends each request — stateless, scales across services, but can't be revoked before expiry. Best practice: short-lived JWTs + refresh tokens (in HttpOnly cookies), plus a Redis blocklist if you need instant logout.

OAuth 2.0 vs OpenID Connect

★ high-freqauth

OAuth 2.0 is an authorization framework — it lets an app access resources on a user's behalf without their password (“let this app post to your Twitter”). OpenID Connect is an authentication layer built on top of OAuth that adds an ID token proving who the user is (“sign in with Google”). Tip: always use the state parameter to prevent CSRF in the flow.

How do you store passwords securely?

★ high-freqauth

Never store plaintext or reversible encryption. Hash with a slow, salted algorithm designed for passwords — bcrypt, scrypt, or Argon2 — with a per-user random salt (defeats rainbow tables) and a high work factor. Add rate limiting / lockout to slow brute force. MD5/SHA-1 are not acceptable.

Session hijacking & fixation

auth

Hijacking — stealing a valid session token (via XSS, sniffing) to impersonate the user; mitigate with HTTPS, HttpOnly/Secure cookies, and rotating tokens. Fixation — forcing a known session ID onto a victim before they log in; mitigate by issuing a fresh session ID on login.

Cryptography

Crypto & Transport

Symmetric vs asymmetric encryption

★ high-freqcrypto

Symmetric uses one shared key for encrypt+decrypt — fast, but key distribution is the problem (AES). Asymmetric uses a public/private key pair — slower, but solves key exchange and enables signatures (RSA, ECC). TLS uses asymmetric to exchange a symmetric session key, then symmetric for the bulk data.

Hashing vs encryption vs encoding

★ high-freqgotchacrypto

A favourite “do you actually get it” question. Encoding (Base64) is reversible and not security — just a format change. Encryption is reversible with a key — for confidentiality. Hashing is one-way — for integrity and password storage; you can't get the input back.

How does HTTPS/TLS keep data safe?

crypto

It gives three things: confidentiality (encryption), integrity (tamper detection), and authentication (the server's certificate, signed by a trusted CA, proves identity). The handshake uses asymmetric crypto to agree a symmetric session key, then encrypts all traffic with it.

Digital signatures & certificates

crypto

A digital signature is data hashed and encrypted with the sender's private key — anyone with the public key can verify integrity and authenticity. A certificate binds a public key to an identity and is signed by a Certificate Authority (CA), forming the chain of trust (PKI) that HTTPS relies on.

HMAC

crypto

A keyed hash (hash + secret key) that verifies both integrity and authenticity of a message — only someone with the key can produce a valid HMAC. Used to sign API requests and JWTs (HS256).

Hardening

Hardening & Headers

CORS — what problem does it solve?

★ high-freqhardening

Browsers block cross-origin JavaScript requests by default (Same-Origin Policy). CORS lets a server opt in via headers like Access-Control-Allow-Origin. For non-simple requests the browser first sends a preflight OPTIONS. Common mistake: Access-Control-Allow-Origin: * on an authenticated API — always name the exact origin.

Content Security Policy (CSP)

hardening

An HTTP header that restricts where scripts/styles/images can load from — the strongest defence against XSS because it can disable inline scripts entirely. e.g. Content-Security-Policy: script-src 'self' blocks scripts from other origins.

Secure cookie attributes

★ high-freqhardening

Harden cookies with HttpOnly (JS can't read it — blocks XSS theft), Secure (HTTPS only), and SameSite Lax/Strict (not sent on cross-site requests — blocks CSRF). Don't store sensitive data directly in cookies.

How would you secure a REST API?

★ high-freqhardening

A common open-ended question — hit the breadth: authentication (OAuth2/JWT/API keys), authorization per endpoint (RBAC, least privilege), HTTPS everywhere, input validation, rate limiting (anti brute-force/DoS), CORS configured tightly, and logging/monitoring for anomalies. An API gateway can enforce much of this.

Principle of least privilege & defense in depth

hardening

Least privilege — every user/service gets the minimum access needed, so a compromise is contained. Defense in depth — layer multiple independent controls (WAF, validation, authz, encryption) so no single failure is fatal.

Measuring

Measuring Performance

Latency vs throughput

★ high-freqbasics

Latency = time for one request (ms). Throughput = requests handled per second. They're distinct and often in tension — batching raises throughput but can raise latency. Optimise for whichever the product needs (a trading system wants low latency; a batch pipeline wants high throughput).

Why p95/p99, not the average?

★ high-freqgotchabasics

Averages hide the slow tail. Percentiles (p50/p95/p99) tell you what the unluckiest users experience — a 50ms average can hide a 2s p99 that's hurting real users. At scale, almost every page touches several services, so a bad tail compounds. Always report tail latency.

How do you find a performance bottleneck?

basics

Measure before optimising. Profile to find where time actually goes (CPU, I/O, DB, network), check metrics/traces and the slow query log, reproduce under load, fix the single biggest contributor, then re-measure. Avoid guessing or premature micro-optimisation.

Caching

Caching

Why and where to cache

★ high-freqcaching

Caching stores results of expensive work (DB reads, computations, rendered pages) closer to the user to cut latency and load. Layers: browser, CDN (static assets), application/in-memory (Redis/Memcached), and database query cache. The first lever for read-heavy systems.

Cache write strategies

★ high-freqcaching

Know these by name: cache-aside (app loads on miss, writes DB then invalidates — most common), write-through (write cache+DB together — consistent, slower writes), write-back (write cache, flush to DB later — fast, risk of loss), write-around (write DB, skip cache — avoids caching rarely-read data).

Eviction policies

caching

When the cache is full: LRU (evict least-recently-used — default for most), LFU (least-frequently-used), TTL (expire after a time), FIFO. Choice depends on access pattern; LRU fits temporal locality.

Cache invalidation — the hard problem

gotchacaching

Keeping cache and source of truth in sync is famously hard (stale data vs cache misses). Strategies: TTL expiry, explicit invalidation on write, and versioned keys. Watch cache stampede (many misses hit the DB at once when a hot key expires — mitigate with locks, request coalescing, or jittered TTLs) and hot keys.

Redis vs Memcached

caching

Both are in-memory caches. Memcached is a simple, multi-threaded key-value store. Redis adds rich data structures (lists, sets, sorted sets), persistence, pub/sub, and clustering — so it doubles as a lightweight datastore, rate limiter, and queue.

Data layer

Database Performance

Speeding up a slow database

★ high-freqdatabase

The usual levers: add the right index (and read the EXPLAIN plan), rewrite queries / avoid SELECT *, add a cache, use read replicas to offload reads, connection pooling to avoid per-request connection cost, and denormalize hot read paths. Then sharding if a single node is the limit.

The N+1 query problem

★ high-freqgotchadatabase

Fetching a list (1 query) then looping to fetch each item's detail (N queries) — a top cause of slow endpoints, common with ORMs. Fix by batching: a single JOIN or an IN (...), or eager-loading the relation.

Offset vs cursor pagination

database

LIMIT/OFFSET is simple but gets slow on deep pages (the DB still scans skipped rows) and can skip/duplicate rows under writes. Cursor (keyset) pagination uses WHERE id > last_id — fast and stable for large, changing datasets.

Read replicas & connection pooling

database

Read replicas copy the primary so reads scale out (accept eventual replication lag). Connection pooling reuses a fixed set of DB connections across requests, avoiding the high cost of opening one per request — essential under load.

Scaling

Scaling & Throughput

Vertical vs horizontal scaling

★ high-freqscaling

Vertical (scale up) — bigger machine; simple but has a ceiling and a single point of failure. Horizontal (scale out) — more machines behind a load balancer; near-unlimited but needs statelessness, coordination, and a data-sharing strategy. Modern systems favour horizontal.

Async processing & message queues

★ high-freqscaling

Move slow or spiky work (emails, image processing, notifications) off the request path into a queue (Kafka, RabbitMQ, SQS) processed by workers. This smooths load spikes (buffering), decouples services, and keeps the user-facing request fast. Add idempotency + retries for reliability.

Rate limiting

scaling

Caps requests per client to protect a service from abuse/overload and ensure fairness. Algorithms: token bucket (allows bursts), leaky bucket (smooths output), fixed/sliding window counters — typically backed by Redis and returning HTTP 429 when exceeded.

Compression & CDNs for throughput

scaling

Gzip/Brotli compress responses to cut bytes over the wire (big win for text/JSON/HTML). A CDN serves static assets from edge nodes near users, cutting latency and origin load. Both are cheap, high-impact wins.

Concurrency

Concurrency & App-level

Concurrency vs parallelism

★ high-freqgotchaapp

Concurrency = dealing with many tasks by interleaving (progress on several, not necessarily at once). Parallelism = literally running tasks simultaneously on multiple cores. Concurrency is about structure; parallelism is about execution. You can have one without the other.

Blocking vs non-blocking (async) I/O

app

Blocking I/O ties up a thread waiting for a response; under load you exhaust the thread pool. Non-blocking/async I/O lets one thread handle many in-flight operations (event loop), giving far higher concurrency for I/O-bound work — the model behind Node.js and modern servers.

Thread pools & backpressure

app

A thread pool reuses a bounded set of worker threads, capping resource use and avoiding the cost of spawning per task. Backpressure is signalling upstream to slow down when a consumer can't keep up (bounded queues, rejecting/shedding load) so the system degrades gracefully instead of collapsing.

The Approach

The framework — how to drive the interview

★ high-freqapproach

Don't jump to boxes. Follow a structure and think out loud:

  • 1. Requirements — clarify functional (what it does) and non-functional (scale, latency, availability, consistency).
  • 2. Estimation — rough QPS, storage, bandwidth so choices are grounded.
  • 3. API — define the key endpoints.
  • 4. Data model — entities, and SQL vs NoSQL choice.
  • 5. High-level design — the boxes: clients, LB, services, DB, cache, queue.
  • 6. Deep dive & bottlenecks — scale the hot path, handle failure, justify trade-offs.

Functional vs non-functional requirements

approach

Functional = features (send a message, shorten a URL). Non-functional = qualities (scalability, availability, latency, consistency, durability). Senior candidates pin down non-functionals early because they drive every architecture decision.

Capacity / back-of-envelope estimation

approach

Translate scale into numbers: daily active users → QPS (and peak QPS), data per record → total storage over N years, read:write ratio (most systems are read-heavy → cache). e.g. “100M URLs/day → ~1.2K writes/s, read-heavy → cache + NoSQL.” Walk the interviewer through the math.

Components

Building Blocks

Load balancing

★ high-freqbuilding-block

Spreads traffic across servers for availability and scale. Algorithms: round-robin, least-connections, IP-hash, weighted. L4 (fast, transport-level) vs L7 (smart, can route by path/header). Health checks remove dead servers automatically.

Caching layers

★ high-freqbuilding-block

Reduce latency and DB load by storing hot data in memory (Redis/Memcached) or at the edge (CDN). Decide what to cache, the eviction policy (LRU), and the invalidation strategy. The first scaling lever for read-heavy systems.

Database replication & sharding

★ high-freqbuilding-block

Replication copies data to replicas for read scaling and failover (primary-replica; mind replication lag). Sharding splits data across nodes by a shard key for write/storage scaling. Together they scale a database horizontally.

Consistent hashing

building-block

Distributes keys across nodes so that adding/removing a node only remaps a small fraction of keys (not everything, as plain hash % N would). The backbone of distributed caches and sharded datastores; virtual nodes smooth the distribution.

Message queues & pub/sub

★ high-freqbuilding-block

Decouple producers from consumers and absorb spikes. Queues (SQS, RabbitMQ) deliver each message to one worker; pub/sub (Kafka, SNS) fans a message out to many subscribers. Enable async processing, buffering, and resilience.

Microservices vs monolith

building-block

Monolith — one deployable; simple to build/test early, harder to scale teams and parts independently. Microservices — independently deployable services; flexible and scalable but add network, data-consistency, and ops complexity. Right answer is contextual, not dogmatic.

Idempotency & the API gateway

building-block

Idempotency (often via an idempotency key) makes retried requests safe — essential for payments and at-least-once queues. An API gateway is the single entry point that handles auth, rate limiting, routing, and aggregation in front of services.

Classic designs

Classic Designs — I

Design a URL shortener (TinyURL)

★ high-freqdesign it

The most common design question. Crux: generate a unique short code by base-62 encoding an auto-increment ID (or hashing + collision check); store the short → long mapping. It's read-heavy, so put a cache in front and pick a key-value/NoSQL store; scale reads with replicas, writes with sharding. Add a rate limiter and 301 redirects. Discuss collision handling and custom aliases.

Design a rate limiter

★ high-freqdesign it

A classic building-block design. Pick an algorithm (token bucket for bursts, sliding window for accuracy), store counters in Redis for shared distributed state, return HTTP 429 on limit, and apply per-user/per-IP/per-tenant rules. Discuss race conditions across instances and where it sits (gateway vs app).

Design a distributed cache

design it

Partition keys across nodes with consistent hashing, choose an eviction policy (LRU/LFU/TTL), and decide write strategy (write-through vs write-back). Cover replication for availability, hot-key handling, and cache-stampede prevention.

Design a unique ID generator

design it

Generate sortable unique IDs at scale without a single bottleneck. Options: a Snowflake-style 64-bit ID (timestamp + machine ID + sequence), DB ticket servers, or UUIDs (random, not sortable). Discuss ordering, clock skew, and coordination.

Design Pastebin

design it

Like URL shortener but stores content. Key insight: keep metadata in the database and the actual text blob in object storage (S3) rather than bloating the DB; add TTL expiry, access control, and a CDN/cache for reads.

Classic designs

Classic Designs — II

Design a chat system (WhatsApp/Messenger)

★ high-freqdesign it

Use WebSockets for real-time bidirectional delivery, a connection/presence service to track who's online, a message store (often a wide-column DB), and a queue for reliability and offline delivery. Cover delivery/read receipts, fan-out for group chats, and message ordering.

Design a news feed (Twitter/Instagram)

★ high-freqdesign it

The core trade-off is fan-out on write (precompute each follower's feed — fast reads, expensive for celebrities) vs fan-out on read (assemble at request time — cheaper writes, slower reads). Most use a hybrid: push for normal users, pull for high-follower accounts. Add caching, pagination, and ranking.

Design a notification system

design it

A queue-driven fan-out: a service publishes events to a message queue, workers deliver across channels (push via APNs/FCM, SMS, email), with retries, user preferences, templating, and rate limiting. Emphasise reliability and idempotent delivery.

Design search autocomplete (typeahead)

design it

Serve prefix suggestions in milliseconds using a trie (often precomputed with top-k results per node) plus heavy caching of popular prefixes, distributed/sharded by prefix. Add debouncing on the client and ranking by popularity.

Design a web crawler

design it

BFS over the web from seed URLs using a URL frontier (prioritised queue), with politeness (robots.txt, rate limits per host), deduplication (bloom filters / checksums), DNS caching, and distributed workers. Handle traps and infinite loops.

Design a file store (Dropbox/Drive)

design it

Split files into chunks for efficient upload/sync, store content in object storage with metadata in a DB, deduplicate identical chunks, sync via a change/notification service, and resolve conflicts. Use a CDN for downloads.

Design video streaming (YouTube/Netflix)

design it

Store source video in object storage, run a transcoding pipeline into multiple resolutions, serve via adaptive bitrate streaming from a global CDN, and keep metadata/recommendations in separate services. Emphasise CDN, encoding, and read-heavy scale.

Design ride-sharing (Uber)

design it

Core pieces: rider + driver services, real-time location tracking, a geospatial index (geohashing / quadtree) for nearby-driver matching, a dispatch service, plus booking/payment and notifications. Discuss matching, surge, and high-write location updates.

Fundamentals

ML & DL Fundamentals

AI vs ML vs Deep Learning vs GenAI

★ high-freqbasics

The standard opener. AI is the broad goal of machines doing tasks that need human-like intelligence. ML is a subset that learns patterns from data instead of hand-coded rules. Deep learning is ML using multi-layer neural networks (great for images, text, audio). Generative AI is DL that creates new content (text, images, code) rather than only predicting/classifying.

Supervised vs unsupervised vs reinforcement learning

★ high-freqbasics

Supervised learns from labelled data (classification, regression). Unsupervised finds structure in unlabelled data (clustering, dimensionality reduction). Reinforcement learns by trial and error via rewards (an agent maximising cumulative reward). Interviewers want you to know when each applies, not just the definitions.

The bias-variance trade-off

★ high-freqbasics

The most-asked ML concept. High bias (too simple) underfits — it misses the real pattern. High variance (too complex) overfits — it memorises noise and fails on new data. Total error = bias² + variance + irreducible error; the goal is the sweet spot that minimises test error, not training error.

Overfitting — how do you prevent it?

★ high-freqbasics

Overfitting = the model learns noise specific to the training set and doesn't generalise. Remedies: more/cleaner data, a simpler model, cross-validation, regularization (L1/L2), dropout (for nets), early stopping, and data augmentation. Name several with the why.

L1 vs L2 regularization

gotchabasics

Both penalise large weights to curb overfitting. L1 (Lasso) adds the sum of absolute weights — drives some weights to exactly zero, so it does feature selection. L2 (Ridge) adds the sum of squared weights — shrinks weights smoothly toward zero without eliminating them.

Gradient descent (and its variants)

★ high-freqtraining

The optimisation that trains most models: compute the loss, take its gradient w.r.t. the weights, step the weights in the negative-gradient direction, repeat. Batch uses all data per step (stable, slow); stochastic (SGD) uses one sample (noisy, fast); mini-batch uses a small batch (the practical default). The learning rate controls step size.

Precision, recall, F1 — and when accuracy lies

★ high-freqgotchaevaluation

Precision = of predicted positives, how many were right (cost of false positives). Recall = of actual positives, how many you caught (cost of false negatives). F1 is their harmonic mean. On imbalanced data (99% one class) accuracy is misleading — a model predicting the majority class scores 99% yet is useless; report precision/recall/F1 or ROC-AUC instead.

Bagging vs boosting

models

Both are ensembles. Bagging trains models in parallel on bootstrapped subsets and averages them to cut variance (Random Forest). Boosting trains models sequentially, each fixing the previous one's errors, to cut bias (XGBoost, LightGBM). Boosting usually wins accuracy; bagging is more robust to noise.

Data leakage

gotchadata

When information unavailable at prediction time sneaks into training — e.g. a feature derived from the target, or scaling/imputing before the train-test split. It produces great offline metrics that collapse in production. Interviewers love handing you a dataset with a deliberately leaky feature to see if you catch it.

Neural nets

Deep Learning

How does a neural network learn? (backprop)

★ high-freqnets

Forward pass: inputs flow through layers to a prediction. A loss function measures the error. Backpropagation sends that error backward, using the chain rule to compute each weight's contribution (gradient), and gradient descent updates the weights. Repeat until it generalises.

Activation functions

★ high-freqnets

They add non-linearity so nets can model complex patterns. ReLU (max(0,x)) is the default — cheap, mitigates vanishing gradients. Sigmoid/tanh squash to a range (used at outputs / older nets) but saturate. Softmax turns logits into a probability distribution for multi-class output. Without non-linearity a deep net collapses to a single linear layer.

Vanishing & exploding gradients

gotchanets

In deep nets gradients can shrink toward zero (early layers stop learning) or blow up (unstable training). Fixes: ReLU activations, batch normalization, residual/skip connections, careful weight init, and gradient clipping (cap the gradient norm — common in RNNs/LSTMs).

Batch normalization & dropout

nets

Batch norm normalises each layer's inputs per mini-batch — stabilises and speeds training. Dropout randomly zeroes a fraction of activations during training so the net can't rely on any single neuron — a regularizer that reduces overfitting (disabled at inference).

CNN vs RNN (and why transformers replaced RNNs)

★ high-freqarchitectures

CNNs use convolutions to capture spatial patterns — images. RNNs/LSTMs process sequences step-by-step with a hidden state — text/time-series, but they're slow (sequential) and struggle with long-range dependencies. Transformers process a whole sequence in parallel via attention, which is why they displaced RNNs for language.

Transfer learning

training

Reuse a model pre-trained on a large dataset as the starting point for a new task, fine-tuning on far less data and compute. It's the backbone of modern practice — from ImageNet-pretrained vision models to foundation LLMs adapted to a domain.

GANs vs diffusion models

generative

Both generate images. GANs pit a generator against a discriminator — fast but unstable to train and prone to mode collapse. Diffusion models learn to denoise from pure noise step by step — slower to sample but higher quality and more stable; they power most modern image generators.

LLM internals

Transformers & LLMs

The transformer architecture

★ high-freqtransformers

The foundation of every modern LLM (“Attention Is All You Need”). It drops recurrence for self-attention, processing all tokens in parallel. Key pieces: token + positional encoding, multi-head self-attention, feed-forward layers, plus residual connections and normalization for stable training.

Self-attention (Q, K, V) — explain it

★ high-freqtransformers

An interview gate. Each token gets three learned vectors: Query, Key, Value. For each token, dot its Query with every Key to get relevance scores, scale by √dk, softmax into weights, and take the weighted sum of Values. This lets every token attend to every other — capturing long-range dependencies without recurrence. The √dk scaling stops large dot products from saturating softmax and killing gradients.

text
Attention(Q, K, V) = softmax( (Q . K^T) / sqrt(d_k) ) . V

Why multi-head attention?

transformers

Instead of one attention computation, several heads run in parallel in different representation subspaces, then concatenate. Each head can focus on a different kind of relationship (syntax, coreference, position), giving a richer view than a single head.

Encoder vs decoder vs encoder-decoder

★ high-freqtransformers

Encoder-only (BERT) uses bidirectional attention — best for understanding (classification, NER, similarity). Decoder-only (GPT, Llama) uses causal masking (each token sees only previous ones) — best for generation. Encoder-decoder (T5) encodes input then decodes output (translation, summarisation). Most 2026 LLMs are decoder-only because generation is the primary use case and scaling laws favour it.

Tokenization & embeddings

transformers

Tokenization splits text into subword tokens (“unbelievable” → un / believ / able) — a balance between vocab size and coverage. Embeddings map tokens to dense vectors where semantic similarity is geometric proximity. Static embeddings (Word2Vec) give a fixed vector per word; transformers produce contextual embeddings that change with surrounding text. (Tokenization is also why models miscount letters — they never see characters.)

Context window & KV cache

★ high-freqinference

The context window is the max tokens (input + output) a model can attend to at once — bigger means more context but more compute (attention is O(n²) in sequence length). The KV cache stores past keys/values during autoregressive generation so each new token doesn't recompute them — essential for fast inference, but it grows with sequence length and dominates memory at long context.

Temperature & decoding strategies

inference

After the model outputs logits, temperature scales them before softmax: low → deterministic/focused, high → diverse/creative. Decoding: greedy (always the top token — can be repetitive), beam search (keeps several candidate sequences — good for translation), top-k / top-p (nucleus) sampling (sample from the most probable tokens — the default for open-ended generation).

Mixture of Experts (MoE)

✨ 2026architectures

A scaling trick used by many 2026 frontier models: replace dense feed-forward layers with many “expert” sub-networks and a router that activates only a few per token. You get the capacity of a huge model while only computing a fraction of it per token — more parameters, similar inference cost.

The modern transformer stack

✨ 2026deeper

Senior signal: today's LLMs refine the 2017 block — RMSNorm instead of LayerNorm, SwiGLU instead of ReLU, RoPE (rotary position embeddings) for better length extrapolation, grouped-query attention (GQA) to shrink the KV cache, and FlashAttention for IO-aware, memory-efficient attention.

Applied GenAI

Building with LLMs

RAG vs fine-tuning vs prompting — when to use which?

★ high-freqgenai

The single most common GenAI design question. The heuristic: start with prompt engineering (change the instructions — cheapest, fastest); escalate to RAG when the gap is knowledge (facts that change or need citations); fine-tune only when the model consistently fails at a style/format/behaviour that prompting and retrieval can't fix. In production, RAG + fine-tuning often combine: fine-tune for tone, retrieve for facts.

What is RAG, and why use it?

★ high-freqgenai

Retrieval-Augmented Generation fetches relevant external documents at query time and feeds them into the prompt so the model answers from real sources. It directly addresses two LLM weaknesses: hallucination (ground answers in retrieved text) and knowledge currency (use data newer than the training cutoff, with citations).

How does a RAG pipeline work?

genai

Index: chunk documents → embed each chunk → store vectors in a vector database. Query: embed the question → retrieve the top-k nearest chunks (often hybrid search = semantic + keyword) → optionally rerank → stuff the best chunks into the prompt → generate. Chunking strategy, embedding model, and retrieval quality make or break it.

LoRA, QLoRA & PEFT

★ high-freqfine-tuning

Full fine-tuning of a billion-parameter model is expensive. PEFT (parameter-efficient fine-tuning) updates only a small set of weights. LoRA freezes the base model and inserts small trainable low-rank matrices into the layers. QLoRA adds quantization so you can fine-tune huge models on a single consumer GPU. The standard way teams customise open models cheaply.

How does RLHF work?

✨ 2026alignment

Reinforcement Learning from Human Feedback aligns a model to human preferences in three steps: (1) supervised fine-tune on demonstrations, (2) train a reward model from human comparisons of outputs, (3) optimise the LLM with PPO to maximise that reward, with a KL penalty to stop it drifting too far and gaming the reward with hollow text. RLAIF / Constitutional AI replace human labels with AI feedback against a set of principles.

Chain-of-Thought & ReAct prompting

★ high-freqprompting

Few-shot gives examples in the prompt. Chain-of-Thought (CoT) asks the model to reason step by step — big gains on math/logic/multi-step tasks because it allocates more computation to the problem. ReAct (Reason + Act) interleaves reasoning with tool calls — the backbone pattern for agents.

Hallucination — what is it and how do you reduce it?

★ high-freqreliability

When a model produces fluent but false or unsupported content, because it predicts plausible tokens, not verified facts. Mitigations: RAG with source grounding and citations, lower temperature, prompt instructions to say “I don't know,” output validation/guardrails, and LLM-as-judge faithfulness checks against the retrieved context.

How do you evaluate a RAG / LLM system?

★ high-freqevaluation

There's no single right answer, so measure dimensions separately. For RAG: context relevance (did retrieval return the right chunks?), faithfulness (is the answer grounded, no hallucination?), and answer relevance (does it address the question?). Automate with LLM-as-judge, plus production signals: cost/token, latency, citation accuracy, and user thumbs-up/down.

Embeddings & vector databases

genai

Embeddings turn text into vectors capturing meaning; a vector database (Pinecone, Weaviate, pgvector, FAISS) indexes them for fast approximate-nearest-neighbour search. This is the retrieval substrate behind RAG and semantic search — find by meaning, not keywords.

Autonomy

Agentic AI

Generative AI vs agentic AI

★ high-freq✨ 2026agents

The framing question of 2026. Generative AI responds to a single prompt and stops. Agentic AI is a loop: it reasons about a goal, plans, calls tools, observes results, and adapts — running multiple steps without a human at each one. An LLM writing code is generative; an agent that runs tests, reads logs, fixes the code, and verifies is agentic.

Tool use / function calling vs a true agent

★ high-freqagents

A common trap. Function calling is a single decision — the model suggests which function to call with which arguments. An agent is the sustained loop wrapped around it: planning, memory, error recovery, and deciding what to do with each result. Function calling is the mechanism; the agent is the orchestration.

The ReAct loop

★ high-freqagents

The default single-agent pattern: the model emits a Thought, picks an Action (tool call), reads the Observation, and repeats until it can answer. Always cap the loop (e.g. 5 steps) and log every step — the classic failure modes are infinite loops and budget blowouts.

text
Thought: I need the user's order status.
Action: lookup_order(order_id="12345")
Observation: Order 12345 shipped on Mar 25.
Thought: I have the answer.
Action: respond("Your order shipped on March 25.")

When do you need a multi-agent system?

agents

Not by default — multi-agent adds real complexity. Justify it when subtasks need genuinely different tools/skills, when isolating failure domains improves reliability, when work can run in parallel, or for generator/critic patterns where one agent reviews another. A controller or workflow graph coordinates the specialists.

Model Context Protocol (MCP)

✨ 2026agents

An open standard (introduced by Anthropic) for exposing tools, data, and resources to LLMs in a uniform way — so any compatible model can discover and call them without bespoke glue. It's become a common vocabulary for agent–tool integration in 2026; the live debate is its token overhead vs direct CLI/API calls for high-volume pipelines.

Agent memory

agents

Agents need state beyond a single call: short-term (the working context / scratchpad for the current task) and long-term (persisted facts and past interactions, usually retrieved from a vector store). Good memory design — what to keep, summarise, or forget — is what keeps multi-step agents coherent without blowing the context window.

Semantic routing

✨ 2026deeper

Using a full LLM just to route a request to the right specialist is slow and costly. The 2026 pattern embeds the query with a fast embedding model and takes cosine similarity against predefined route embeddings (“zero-shot classification”) in <10ms — falling back to an LLM only when the similarity is below threshold.

Agent failure modes & guardrails

★ high-freqgotchasafety

Agents fail in ways chatbots don't: infinite Thought–Action loops, runaway cost, calling the wrong/hallucinated tool, and — critically — prompt injection, which in an agent with tool access can escalate to remote code execution or data exfiltration. Defences: step/budget caps, output/schema validation, tool allow-lists and permissions, sandboxing, human-in-the-loop approval gates, and full trajectory logging.

Shipping it

Production, Cost & Safety

LLM cost & latency — what drives them?

★ high-freqproduction

APIs charge per token, so sending 4,000 tokens when 1,500 would do costs ~2.6× more. Latency is dominated by time-to-first-token and then per-token generation; huge contexts raise both (and money). Levers: trim prompts, cap output, cache, route to smaller models, and stream the response.

Making models cheaper/faster: quantization, distillation, pruning

production

Quantization stores weights at lower precision (e.g. 8-bit/4-bit) — less memory, faster, minor quality loss. Distillation trains a small “student” to mimic a large “teacher.” Pruning removes unimportant weights. Plus speculative decoding (a small model drafts, the big one verifies) to cut latency.

Serving LLMs at scale

production

Throughput comes from batching many requests and from memory-efficient KV-cache management — PagedAttention (vLLM) is the standard technique. Model routing / LLM gateways send easy sub-tasks to small cheap models and hard ones to frontier models; prompt/semantic caching avoids re-answering repeats.

LLMOps & observability

production

Because outputs are non-deterministic, you instrument everything: prompt/response logging, token cost and latency metrics, eval suites run on every change, drift monitoring, and guardrails on inputs and outputs. “It worked in the demo” isn't production — you need traces to debug and evals to ship safely.

Prompt injection & guardrails

★ high-freqsafety

The top LLM security risk: untrusted input that overrides your instructions (“ignore previous instructions…”), especially dangerous when the model has tools or sees retrieved/web content (indirect injection). Defences: separate trusted vs untrusted content, input/output filtering, least-privilege tools, sandboxing, and human approval for high-impact actions.

Responsible AI & explainability

✨ 2026governance

Expect at least one. Explainability is hard because LLMs use billions of opaque parameters — practical answers come from system design: RAG source attribution, citations, audit logs, confidence signals, and model cards. Know the governance landscape: the EU AI Act (risk-based, phasing in across 2026–27), ISO/IEC 42001 (AI management standard), and risks like deepfakes needing human-in-the-loop verification.

Latest & greatest // 2026

The 2026 Frontier

Reasoning models & test-time compute

★ high-freq✨ 2026frontier

The defining shift: models like OpenAI's o-series and DeepSeek-R1 are trained (with RL) to produce long internal chains of thought, spending more compute at inference to reason through hard problems. The lever moved from “bigger model” to “let it think longer.” Interviewers ask about the accuracy-vs-latency/cost trade-off and when reasoning models are worth it.

The agentic shift

✨ 2026frontier

The industry's centre of gravity in 2026 moved from chat to autonomous agents — systems that plan and act across tools (coding agents, research agents, computer-use agents). The AI-engineer role is reframing from “prompt engineer” to “agentic system architect,” and nearly every AI interview now includes an agent question. Anthropic's “Building Effective Agents” is the most-cited reference.

Long context vs RAG

✨ 2026gotchafrontier

Context windows have stretched to ~1–2M tokens, but bigger isn't free: stuffing a million tokens costs dollars per call, pushes time-to-first-token over 30s, and attention can still degrade amid irrelevant data. So the 2026 pattern isn't “just use a huge context” — it's targeted retrieval (multi-agent RAG, semantic routing) to feed the model a concise, relevant prompt.

Multimodal models

✨ 2026frontier

The transformer is now a universal backbone: vision (ViT), audio (Whisper), and text tokens share one latent space, so a single model can see, hear, and read (e.g. GPT-4o-style). Expect questions on unifying modalities into a common token space and the applications it unlocks.

Beyond attention: Mamba & state-space models

✨ 2026research

Standard self-attention is O(n²) in sequence length. State-space models (SSMs) like Mamba, and various linear-attention schemes, aim for near-linear scaling on long sequences — an active research frontier and an increasingly common “do you follow the field?” probe.

Mixture-of-Experts at frontier scale

✨ 2026frontier

Most 2026 frontier models are sparse MoE: trillions of parameters in total, but a router activates only a few experts per token, so inference stays affordable. It's how labs keep scaling capacity without scaling cost linearly — worth being able to explain the router + sparse-activation idea.

Small & on-device models

✨ 2026frontier

Counter-trend to giant models: highly capable small models (distilled/quantized) now run on laptops and phones, enabling private, low-latency, offline inference. Production systems increasingly route — small models for easy steps, frontier models for hard ones — to balance cost, latency, and quality.

AI for science & GraphRAG

✨ 2026applied

Two fast-moving applied frontiers worth a mention: transformer models for science (e.g. AlphaFold-style protein/molecule structure prediction accelerating drug discovery), and GraphRAG, which builds a knowledge graph over your corpus so retrieval can follow relationships and answer multi-hop questions a flat vector search would miss.
