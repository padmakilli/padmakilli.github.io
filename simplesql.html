DDL · Data Definition Language

Defining structure

DDL statements create and change the shape of your database — databases, tables, columns, constraints, indexes, and views. They define the container; DML (next) fills it with data.

CREATE DATABASE

Creates a brand-new, empty database.

A database is the top-level container for tables and other objects. You then connect to it (\c company_db in psql) before creating tables inside it.

Common data types

Every column has a type. Choosing the right one enforces correctness and saves space.

INT / BIGINTWhole numbers. BIGINT for very large counts/IDs.
SERIAL / BIGSERIALAuto-incrementing integer — perfect for surrogate primary keys.
NUMERIC(p,s)Exact decimals (money!). NUMERIC(10,2) = up to 10 digits, 2 after the point.
REAL / DOUBLE PRECISIONApproximate floating-point — scientific data, not money.
VARCHAR(n) / TEXTStrings. VARCHAR(n) caps length; TEXT is unlimited.
BOOLEANTRUE / FALSE / NULL.
DATE / TIMESTAMPA calendar date, or a date-and-time. TIMESTAMPTZ stores the time zone too.

JSON / JSONBStructured documents. JSONB is binary, indexable, and faster to query.

CREATE TABLE + constraints

Defines a table: its columns, their types, and the rules (constraints) their values must obey.

The constraint keywords each enforce one rule: PRIMARY KEY = unique row identifier, NOT NULL = value required, UNIQUE = no duplicates, CHECK = custom condition, DEFAULT = fallback value, REFERENCES = must match a row in another table (a foreign key).

SERIAL vs GENERATED IDENTITY

Two ways to auto-generate IDs. IDENTITY is the modern SQL-standard approach.

Both produce auto-incrementing integers. Prefer GENERATED ALWAYS AS IDENTITY for new tables — it is clearer about ownership of the sequence and is the cross-database standard.

ALTER TABLE

Changes an existing table's structure without recreating it.

Use ALTER TABLE as requirements evolve. Each clause targets one change: add/drop columns, retype, rename, or attach/detach constraints.

DROP & TRUNCATE

Remove a whole table, or empty it out fast.

DROP destroys the table structure and its data. TRUNCATE keeps the structure but removes every row — far faster than DELETE for wiping a table because it does not scan rows one by one. Add IF EXISTS to avoid errors when the object may not be there.

CREATE INDEX

Builds a lookup structure so the database can find rows quickly.

Without an index, finding rows means scanning the whole table. An index (usually a B-tree) is like a book's index — it turns a full read into a direct jump. Index columns you filter or join on often. The trade-off: indexes speed up reads but slightly slow down writes and use disk.

CREATE VIEW

Saves a query under a name so you can treat it like a virtual table.

A view stores no data of its own — it re-runs its underlying query each time. It's ideal for hiding complex joins behind a simple name, or exposing a limited slice of data. A MATERIALIZED VIEW (below) is the opposite: it caches results.

MATERIALIZED VIEW

Like a view, but it physically stores the results for fast repeated reads.

A regular view is always live but recomputed on every read. A materialized view is fast to read but can go stale — you call REFRESH to update it. Great for expensive dashboards/reports that don't need up-to-the-second numbers.

DML · Data Manipulation Language

Adding & changing data

DML statements work on the rows inside tables: INSERT adds them, UPDATE changes them, DELETE removes them. Postgres's RETURNING clause lets each one hand back the rows it touched.

INSERT

Adds one or more new rows to a table.

Always list columns explicitly — it protects you if the table structure changes later. The server reply INSERT 0 2 means two rows were inserted.

INSERT ... SELECT

Copies rows from a query straight into a table.

Instead of literal VALUES, you feed INSERT the output of a SELECT. Perfect for archiving, backfilling, or copying subsets between tables.

RETURNING Postgres feature

Get back the rows an INSERT/UPDATE/DELETE just affected — including generated IDs.

Without RETURNING you'd need a second query to learn the new auto-generated emp_id. This gives it to you in one round trip — a big win in application code.

INSERT ... ON CONFLICT (upsert)

Insert a row, but if it clashes with an existing one, update instead. "Update-or-insert" = upsert.

When a row with dept_id = 10 already exists, the insert would normally fail on the primary-key conflict. ON CONFLICT catches that: DO UPDATE merges the new values (EXCLUDED refers to the row you tried to insert), or DO NOTHING silently skips it.

UPDATE

Changes values in existing rows. Almost always with a WHERE.

Danger: an UPDATE with no WHERE changes every row in the table. The same is true of DELETE. Always double-check your WHERE clause.

UPDATE ... FROM

Update rows using values looked up from another table.

The FROM clause joins another table into the update so its columns can drive the change — Postgres's clean way to do "update based on a join".

DELETE

Removes rows that match a condition.

DELETE removes whole rows (to blank out a single field, use UPDATE ... SET col = NULL instead). To empty an entire table quickly, prefer TRUNCATE.

DQL · Data Query Language

SELECT — asking questions

SELECT is the heart of SQL and the one you'll write most. It reads data without changing it. Master its clauses — WHERE, ORDER BY, GROUP BY, HAVING — and you can answer almost any question about your data.

SELECT basics

Choose columns, rename them, and compute new ones.

SELECT * grabs all columns; naming them is clearer and faster. AS gives a column a friendly alias. You can compute values (salary * 12) and concatenate text with ||.

WHERE — filtering rows

Keep only the rows that meet a condition.

Comparison operators: =  <> (not equal)  <  >  <=  >=. Combine conditions with AND, OR, and NOT.

BETWEEN · IN · LIKE · IS NULL

Expressive filters for ranges, lists, patterns, and missing values.

NULL means "unknown", so manager_id = NULL is never true — you must write IS NULL / IS NOT NULL. LIKE patterns use % (zero or more characters) and _ (exactly one).

ORDER BY · LIMIT · OFFSET

Sort the results and take just a slice (the basis of pagination).

ORDER BY col DESC sorts descending; add more columns for tie-breakers. LIMIT caps the row count; OFFSET skips ahead — together they page through large result sets.

DISTINCT

Removes duplicate rows from the result.

Eight employees, but only four distinct department values. DISTINCT collapses repeats. Postgres also has DISTINCT ON (col) to keep the first row per group.

Aggregate functions

COUNT, SUM, AVG, MIN, MAX — collapse many rows into one summary number.

Aggregates turn a set of rows into a single value. COUNT(*) counts rows; COUNT(col) ignores NULLs in that column. They're most powerful with GROUP BY.

GROUP BY

Split rows into groups and aggregate each group separately.

One output row per group. The rule: every column in SELECT must either appear in GROUP BY or be wrapped in an aggregate. Here we get per-department headcount and average pay.

HAVING vs WHERE

Filter groups after aggregation. (WHERE filters rows before.)

Key distinction: WHERE can't use aggregates because it runs before grouping. To filter on COUNT(), SUM(), etc., you must use HAVING, which runs after.

CASE — if/then in SQL

Produce different values per row based on conditions.

CASE is SQL's if/else. It checks each WHEN in order and returns the first match's value, falling back to ELSE. Useful for bucketing, labelling, and conditional aggregation.

COALESCE & NULLIF

Handle missing values gracefully.

COALESCE(a, b, ...) returns the first non-NULL argument — perfect for default values. NULLIF(a, b) returns NULL when a = b, a classic trick to sidestep division by zero.

Relational power · combining tables

JOINs

Data is split across tables to avoid repetition. Joins stitch them back together on a shared column — here, matching employees.dept_id to departments.dept_id. The join type decides what happens to rows that have no match.

The four main joins at a glance

INNER JOIN

Returns only rows that have a match in both tables.

Karan (no department) and HR (no employees) both vanish, because INNER JOIN keeps only matched pairs. Table aliases (e, d) keep the query short and unambiguous.

LEFT JOIN

Keeps every row from the left table, filling NULL where the right has no match.

This is the most-used join. All 8 employees appear; Karan's department shows NULL because he has none. Great for "show everything, matched or not" — and for finding unmatched rows by adding WHERE d.dept_id IS NULL.

RIGHT & FULL OUTER JOIN

Mirror image of LEFT, and the union of both.

RIGHT JOIN keeps all rows from the right table (so HR, with no staff, shows up with a NULL employee). FULL OUTER JOIN keeps unmatched rows from both sides — Karan (no dept) and HR (no staff) both appear.

SELF JOIN

Join a table to itself — here, matching each employee to their manager.

The same table is referenced twice with two aliases (e = employee, m = manager). This resolves hierarchies stored in one table. Using LEFT JOIN keeps top-level people (managers with no manager) in the result.

Joining 3+ tables

Chain joins to combine employees, projects, and the junction table.

Many-to-many relationships (employees ↔ projects) are resolved through a junction table. You join across all three, matching each key in turn. (JOIN with no prefix means INNER JOIN.)

Queries inside queries

Subqueries & CTEs

A subquery is a SELECT nested inside another statement. CTEs (Common Table Expressions, the WITH clause) do the same job but read top-to-bottom, making complex logic far clearer.

Subquery in WHERE

Use one query's result to filter another.

The inner query computes one value (the average, 68125), and the outer query compares against it. This "scalar subquery" runs first, then feeds the WHERE.

IN · EXISTS · correlated subqueries

Test membership, or check "does a related row exist?"

IN (subquery) checks whether a value is in a returned list. EXISTS checks only whether the inner query returns any row (it stops at the first). A correlated subquery runs once per outer row because it depends on that row's values.

WITH — Common Table Expression

Name a subquery up front, then use it like a table. Cleaner than nesting.

The WITH block defines dept_avg once; the main query joins to it by name. CTEs make multi-step logic readable and let you reference the same intermediate result multiple times.

WITH RECURSIVE

A CTE that refers to itself — for hierarchies and sequences.

A recursive CTE has two parts joined by UNION ALL: an anchor (the starting rows) and a recursive member that references the CTE itself, running repeatedly until no new rows appear. This is how you traverse org charts, category trees, and graphs.

Combining result sets vertically

Set operations

Where joins combine tables side by side (more columns), set operations stack query results on top of each other (more rows). Both queries must return the same number and types of columns.

UNION · INTERSECT · EXCEPT

Combine, intersect, or subtract two result sets.

UNION merges and removes duplicates; UNION ALL keeps every row (use it when you know there are no dupes — it skips the sort). INTERSECT keeps only rows in both. EXCEPT subtracts the second set from the first — here, finding departments (like HR, 40) that have no employees.

Advanced analytics · rankings & running totals

Window functions

A window function computes across a set of rows related to the current row — but unlike GROUP BY, it does not collapse them. You keep every row and get an extra analytical column: a rank, a running total, the previous row's value.

ROW_NUMBER · RANK · DENSE_RANK

Number or rank rows within partitions.

OVER (PARTITION BY dept_id ...) restarts the ranking for each department. ROW_NUMBER always gives unique 1,2,3; RANK leaves gaps after ties (1,1,3); DENSE_RANK doesn't (1,1,2). Classic use: "top 3 earners per department".

Running totals & LAG/LEAD

Aggregate cumulatively, or peek at neighbouring rows.

An aggregate like SUM() with an OVER (ORDER BY ...) becomes a running total — it accumulates down the rows instead of collapsing them. LAG fetches a value from a previous row (and LEAD from a following one), ideal for period-over-period comparisons.

TCL · Transaction Control Language

Transactions

A transaction groups several statements into one all-or-nothing unit. Either every change commits together, or none do. This is what keeps a bank transfer from debiting one account without crediting the other.

BEGIN · COMMIT · ROLLBACK

Start a transaction, then either save or undo all its work.

Between BEGIN and COMMIT, changes are provisional and invisible to others. COMMIT makes them permanent atomically; ROLLBACK discards them entirely. If the money leaves account 1 but the second update fails, you roll back and nothing changed — that's atomicity.

SAVEPOINT

A checkpoint you can roll back to without abandoning the whole transaction.

Savepoints give you partial rollback inside a transaction. ROLLBACK TO sp1 rewinds to the checkpoint without throwing away everything before it — useful for retrying part of a complex operation.

DCL · Data Control Language

Permissions

DCL controls who can do what. You create roles (users/groups) and then grant or revoke privileges on database objects — the foundation of database security.

CREATE ROLE · GRANT · REVOKE

Define a user and control their access.

GRANT gives privileges (SELECT, INSERT, UPDATE, DELETE, and more) on an object to a role; REVOKE removes them. Roles can also be granted to other roles to build groups. Follow the principle of least privilege: give each role only what it needs.

Beyond the basics · why teams pick Postgres

PostgreSQL extras

A few features that make PostgreSQL stand out — native arrays, first-class JSON, and tools to understand query performance.

Arrays

A column can hold a list of values.

Arrays store multiple values in one field. ANY(tags) tests membership, and you can index elements with tags[1] (Postgres arrays are 1-based).

JSON / JSONB

Store and query structured documents natively.

JSONB stores JSON in a binary, indexable form. Operators: -> gets a JSON value, ->> gets it as text, and @> tests containment. This blends relational and document-style data in one place.

EXPLAIN ANALYZE

See exactly how Postgres will run a query — and how long it took.

The query planner picks a strategy (sequential scan, index scan, hash join, ...). EXPLAIN shows the plan and its estimated cost; adding ANALYZE actually runs it and reports real timings. If you see a slow Seq Scan on a big table, that's often a sign you need an index.

Type casting with ::

Convert a value from one type to another.

value::TYPE is Postgres's shorthand for the standard CAST(value AS TYPE). You'll use it constantly — formatting numbers as text, parsing dates, or aligning types so an operation works.

Concepts · the vocabulary behind the syntax

DBMS glossary

The important terms every database developer should know, in plain English — the ideas that turn SQL syntax into real understanding. Use the search box to jump to any term.

Core & structural

Database

An organized collection of data.

A structured set of related data stored electronically so it can be easily accessed, managed, and updated.

DBMS

Database Management System.

The software that stores, retrieves, and manages databases (e.g. PostgreSQL, MySQL, Oracle). It sits between users and the raw data files.

RDBMS

Relational DBMS.

A DBMS that organizes data into tables (relations) with rows and columns, and lets you relate tables via keys. PostgreSQL is an RDBMS.

Table (Relation)

A grid of rows and columns.

The basic storage structure in a relational database. Each table holds data about one type of entity, such as employees.

Row (Tuple / Record)

One entry in a table.

A single horizontal record representing one instance of the entity — e.g. one employee, with a value in each column.

Column (Attribute / Field)

One property in a table.

A vertical slice holding one kind of value for every row, such as salary or email. Each column has a defined data type.

Schema

The blueprint / namespace.

Two senses: (1) the overall design of a database — its tables, columns, and relationships; (2) in Postgres, a named container that groups objects (like public).

Domain

The allowed values for a column.

The set of valid values a column may hold, defined by its data type and constraints — e.g. the domain of salary is positive numbers.

Metadata / Catalog

Data about the data.

Information describing the database's own structure — table names, column types, constraints. Postgres exposes it via the system catalog and information_schema.

Keys

Primary Key

The unique row identifier.

A column (or set of columns) that uniquely identifies each row. It cannot be NULL and cannot repeat. Every table should have one.

Foreign Key

A link to another table.

A column whose values must match a primary key in another table, enforcing a relationship — e.g. employees.dept_id references departments.dept_id.

Candidate Key

A possible primary key.

Any minimal column set that could uniquely identify rows. One candidate is chosen as the primary key; the rest become alternate keys.

Super Key

Any unique-identifying set.

Any set of columns that uniquely identifies a row — including redundant ones. Candidate keys are the minimal super keys.

Composite Key

A key made of several columns.

A primary key spanning two or more columns, used when no single column is unique — e.g. (emp_id, project_id) in the junction table.

Alternate / Unique Key

A candidate key not chosen as primary.

Enforces uniqueness without being the primary key. In SQL you declare it with the UNIQUE constraint (e.g. email).

Surrogate vs Natural Key

Made-up ID vs real-world value.

A surrogate key is a system-generated ID (like a SERIAL) with no business meaning. A natural key is real data that's already unique (like an email or ISBN).

Constraints & integrity

Constraint

A rule enforced on data.

A condition the database guarantees will always hold: NOT NULL, UNIQUE, CHECK, DEFAULT, PRIMARY KEY, FOREIGN KEY.

Entity Integrity

Every row is identifiable.

The rule that a primary key must be unique and never NULL, so each row can be told apart from every other.

Referential Integrity

No broken links.

The rule that every foreign-key value must point to a real row in the referenced table (or be NULL). You can't reference a department that doesn't exist.

Domain Integrity

Valid values only.

Ensures column values fall within their allowed type and constraints — you can't put text in an INT column or a negative salary past a CHECK.

Cascading Actions

Changes ripple through links.

ON DELETE CASCADE deletes child rows when a parent is removed; SET NULL nulls the link; RESTRICT blocks the delete. Controls what happens to related rows.

Design & modeling

Entity

A real-world thing you store.

An object or concept the database tracks — an employee, a department, a project. Entities typically become tables.

Relationship

How entities connect.

An association between entities, such as "an employee belongs to a department". Modeled with foreign keys or junction tables.

ER Model / ER Diagram

A visual data blueprint.

The Entity-Relationship model diagrams entities, their attributes, and the relationships between them — the standard way to design a schema before building it.

Cardinality

How many relate to how many.

The nature of a relationship: one-to-one (1:1), one-to-many (1:N), or many-to-many (M:N). Employees-to-department is 1:N; employees-to-projects is M:N.

Junction Table

Resolves many-to-many.

A bridge table that connects two entities in an M:N relationship, holding a foreign key to each — e.g. employee_projects.

Normalization

Normalization

Organizing tables to reduce redundancy.

The process of structuring tables so data isn't repeated and relationships are clean, which prevents update/insert/delete anomalies. Achieved by applying successive "normal forms". The trade-off is more tables and more joins.

Functional Dependency

One column determines another.

When column A's value fixes column B's value (written A → B). Normalization is built on analyzing these dependencies.

1NF (First Normal Form)

Atomic values, no repeating groups.

Each cell holds a single value and each row is unique. No lists crammed into one field.

2NF (Second Normal Form)

No partial dependencies.

Is in 1NF, and every non-key column depends on the whole composite key, not just part of it.

3NF (Third Normal Form)

No transitive dependencies.

Is in 2NF, and non-key columns depend only on the key — not on other non-key columns. The common practical target.

BCNF

A stricter 3NF.

Boyce-Codd Normal Form: for every dependency A → B, A must be a super key. Resolves edge cases 3NF misses.

Denormalization

Deliberately adding redundancy.

Intentionally duplicating data (fewer joins) to speed up reads — a conscious performance trade-off, common in analytics and warehouses.

Anomalies

Problems from bad structure.

Update, insertion, and deletion anomalies — inconsistencies that arise when data is redundantly stored. Normalization exists to eliminate them.

Transactions & concurrency

Transaction

An all-or-nothing unit of work.

A sequence of operations treated as one logical action — it either fully completes (commits) or fully undoes (rolls back).

ACID Properties

The four guarantees of a reliable transaction.

Atomicity — all steps happen or none do. Consistency — the database moves from one valid state to another, respecting all rules. Isolation — concurrent transactions don't interfere with each other. Durability — once committed, changes survive crashes and power loss.

Concurrency Control

Managing simultaneous access.

Techniques that let many users read and write at once without corrupting data or seeing inconsistent results.

Locking

Reserving data during use.

A mechanism that restricts access to data while it's being changed. Shared (read) locks coexist; exclusive (write) locks don't.

Deadlock

Two transactions stuck waiting.

When transaction A holds a lock B needs, and B holds one A needs — neither can proceed. The DBMS detects this and aborts one.

Isolation Levels

How much transactions see of each other.

From Read Uncommitted to Serializable, they trade concurrency for consistency, controlling phenomena like dirty reads, non-repeatable reads, and phantom reads.

Dirty / Phantom Read

Concurrency side effects.

A dirty read sees another transaction's uncommitted change; a non-repeatable read gets different values for the same row twice; a phantom read sees new rows appear mid-transaction.

MVCC

Multi-Version Concurrency Control.

Postgres's approach: each transaction sees a consistent snapshot, and writers create new row versions instead of blocking readers. Readers never block writers and vice-versa.

Database objects

Index

A fast-lookup structure.

A sorted data structure (usually a B-tree) that speeds up searches on a column, at the cost of extra storage and slightly slower writes.

View

A saved query as a virtual table.

A named SELECT you can query like a table. It stores no data itself and always reflects the latest underlying rows.

Materialized View

A cached query result.

Like a view but physically stores its results for fast reads; must be refreshed to reflect new data.

Stored Procedure / Function

Reusable server-side code.

Precompiled SQL (and procedural logic) saved in the database and invoked by name — for encapsulating business logic close to the data.

Trigger

Code that fires on an event.

A function that runs automatically before or after an INSERT, UPDATE, or DELETE — used for auditing, validation, or derived values.

Cursor

Row-by-row result processing.

A pointer that lets you fetch and process query results one row at a time, instead of all at once — useful for very large result sets.

Sequence

An auto-incrementing counter.

A database object that generates a stream of unique numbers, powering SERIAL and identity columns.

Query language

SQL sublanguages

The five families of SQL commands.

DDL (Definition: CREATE, ALTER, DROP) shapes structure. DML (Manipulation: INSERT, UPDATE, DELETE) changes data. DQL (Query: SELECT) reads it. DCL (Control: GRANT, REVOKE) manages permissions. TCL (Transaction: COMMIT, ROLLBACK) manages transactions.

CRUD

Create, Read, Update, Delete.

The four basic data operations, mapping to INSERT, SELECT, UPDATE, and DELETE — the backbone of almost every application.

Join

Combining rows across tables.

An operation that matches rows from two or more tables on a related column to produce a combined result set.

Aggregate Function

Many rows → one value.

Functions like COUNT, SUM, AVG, MIN, MAX that summarize a group of rows into a single result.

Query Optimizer

Picks the fastest execution plan.

The DBMS component that analyzes a query and chooses how to run it (which indexes, join order, scan types) to minimize cost. Inspect its choices with EXPLAIN.

Execution Plan

The chosen steps to run a query.

The sequence of operations the optimizer settles on — scans, joins, sorts. Reading it is key to diagnosing slow queries.

Architecture & scaling

OLTP vs OLAP

Transactions vs analytics.

OLTP (Online Transaction Processing) handles many small, fast reads/writes — the app database. OLAP (Online Analytical Processing) runs large, complex read queries for reporting and analysis.

Data Warehouse

A store built for analytics.

A large repository consolidating data from many sources, optimized for OLAP-style querying and business intelligence rather than day-to-day transactions.

Replication

Copying data across servers.

Keeping synchronized copies of a database on multiple machines for high availability and read scaling. A primary accepts writes; replicas serve reads.

Partitioning

Splitting one big table.

Dividing a large table into smaller physical pieces (by range, list, or hash) to improve manageability and query performance, while it still looks like one table.

Sharding

Splitting data across databases.

Horizontally partitioning data across multiple independent database servers so each holds a subset — a strategy for scaling beyond one machine.

CAP Theorem

Pick two of three under partition.

A distributed system can't simultaneously guarantee Consistency, Availability, and Partition tolerance — when the network splits, you must trade consistency against availability.
